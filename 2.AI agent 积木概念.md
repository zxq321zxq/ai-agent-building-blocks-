# 积木概念

> 积木 = 独立最小知识单元。7 role 平等。
>
> **7 role 不只是内容的分类——是整个系统的组织语言。** 从"积木怎么存"（知识存储/文件层级）到"积木怎么串"（Skill写作/认知供给/编排引擎）到"积木怎么验"（拓扑依赖/约束对验证/门禁标记）到"积木怎么进化"（知识复利闭环/学习模式），全链路用同一套 7 role 描述。12 个应用面本身也可用 7 role 解释。

## 先看一个完整的积木（定义 role）

```markdown
---
domain: 会计
子域: 进项
role: 定义
scope: step
triggers: [进项, 进项税额, 专票]
confirmed_by: [{source: 晴, weight: 1.0, date: "2025-04-03"}]
---

## compiled_truth

### 定义:进项税额

**场景:** 一般纳税人购进货物/服务时取得增值税专用发票
**概念:** 进项税额 = 购入价 ÷ (1+税率) × 税率。可以抵扣销项税额。
**适用范围:** 一般纳税人。小规模纳税人取得专票不得抵扣，全额计入成本。
```

这就是一个积木。四段 frontmatter 声明身份和信任链，`## compiled_truth` 下是正文，`### {role}:{subject}` 是 role header。步骤类用 `### {do_type}{role}:{subject}`（如 `### 做账步骤:进项存货`）。

**各 role 完整模板见 wiki skill**（定义/步骤/规则/例子/映射/配置/编排各一模板）。下方仅给一个定义 role 的完整样例，其余 6 role 的格式按 wiki skill 模板。

### Skill 引用积木示例

```yaml
# Skill 文件 frontmatter
bricks:
  - 会计/进项/定义/进项税额.md
  - 会计/进项/规则/票种识别.md
  - 会计/进项/步骤/做账-进项存货.md
```

`bricks:` 路径相对于 `wiki/知识库/`，顺序即拼装顺序。cognition_loop 按此列表取积木后按场景排序注入。

### do_type 枚举

步骤 role 的 header 可以带 `do_type` 前缀，标识此步骤属于哪种操作类型：

| do_type | 含义 | 示例 header |
|---------|------|------------|
| 做账 | 会计做账流程 | `### 做账步骤:进项存货` |
| 提取 | 数据提取流程 | `### 提取步骤:销项发票` |
| 审核 | 财务审核流程 | `### 审核步骤:应付账款` |
| 迁移 | 工作台迁移操作 | `### 迁移步骤:六项审计` |
| 创建 | 新建资源操作 | `### 创建步骤:新积木` |

列表开放——新增操作类型时创建新的 do_type 前缀。cognition_loop 按 do_type 过滤匹配步骤。

## 编排 role vs 编排 domain

编排既是一个 **role**（信息形态），也是一个 **domain**（知识域）。区分线：

- `{domain}/编排/` 路径下：编排作为 role——描述此 domain 下的可执行流（如 `会计/编排/做账全流程.md`）
- `编排/` 作为 domain：存放跨 domain 的编排积木——会计管线（提取/分析/审核/做账）的编排文件集中管理在此

规则：单 domain 内的流 → `{domain}/编排/xxx.md`。跨 domain 的组合流 → `编排/{子域}/编排/xxx.md`。

## 核心定义

**7 role = 七种平等的信息形态。** 编排不是元层——和其他 6 role 同级。

| role | 信息形态 | 回答的问题 |
|------|------|------|
| 编排 | 组件 + 流 + 失败 + 参数 + 触发 + 理由 + 局限 | "怎么串" |
| 定义 | 是什么 / 管什么 / 不管什么 | "是什么" |
| 例子 | 具体实例 / 陷阱 | "发生过什么" |
| 规则 | 约束（🛑阻断 / ⚠️域级 / 📎参考） | "不能破什么" |
| 步骤 | 操作内容 | "怎么做" |
| 配置 | 参数 / 依赖 / 自救 | "用什么" |
| 映射 | 参考 / 引用 / 关联 | "去哪查" |

### 单一主张原则

每个积木只包含一个明确的主张或知识单元。不是7 role定义之外的额外约束——是验证粒度的要求。

**原因：** 如果一个积木含多条声明——六网内容级验证中能验证的、不能验证的、通过验证的、验证失败的全部混在一起——reliability失去语义。单一主张保证每条积木在六网中有且仅有一个清晰的验证路径。

**检验标准：** 如果删除积木中任何一段，剩余内容仍能独立成立→违反单一主张→需拆分。

**🛑门禁是规则的分级标记，不是第 8 role。** 在 Skill 文件中可单独成段。

**编排不是元层。** 编排描述"组件怎么组合成可执行的流"（7 字段：组件/流/失败/可变参数/触发/设计理由/已知局限），和其他 6 role 同级——不是高于它们。

**Skill = 积木的排序组合。** 先拆积木（读原版→逐段识别每段属于哪个 role）→统计实际 role 子集→自然归入某一类。类型是拆完后的观察结果，不是预设模板。

## 四场景 role 排序

| 位置 | 0.元原意识 | 1.Agent注入 | 2.知识积木 | 3.Skill文件 |
|:---:|------|------|------|------|
| 1 | 定义 | 编排 | 编排 | 编排 |
| 2 | 规则 | 定义 | 定义 | 定义 |
| 3 | 映射 | 规则 | 例子 | 🛑门禁 |
| 4 | 配置 | 例子 | 规则 | 例子 |
| 5 | — | 步骤 | 步骤 | 规则 |
| 6 | — | 配置 | 配置 | 步骤 |
| 7 | — | 映射 | 映射 | 配置 |
| 8 | — | — | — | 映射 |

**场景 0 为什么只有 4 个 role：** 元原意识层是 Agent 的自我认知——"我是谁、边界在哪"，还没有具体任务和步骤上下文。编排（"现在在哪一步"）需要任务上下文才能回答，步骤（"怎么做"）需要当前操作才能描述，例子（"上次怎么踩的"）需要具体场景才有意义。所以只保留定义（我是谁）、规则（不能破什么）、映射（去哪查）、配置（参数是什么）。此场景不改。

**场景 1 vs 2 唯一差异**：规则和例子互换。Agent 先看边界（约束优先），人先看实例（现象优先）。但 Agent 注入（场景 1）不是按 role 线性排列——是按 concept cluster 空间聚合。同一个概念下的定义、规则、例子紧邻排列，形成自足的概念簇。约束在它要约束的概念旁边，不是在整个上下文的前面。因为 Transformer 自注意力是全局跳跃的，落到同一个概念簇内比跨越概念簇的同类 role token 更高效。元意识层（场景 0）的 4 个 role 是跨概念的——线性排列正好。

**场景 3 = 场景 2 + 🛑门禁从规则中抽出前置**（阻断级不能藏在一堆约束中间）。8行含门禁标记——不是8种role，是7种role+门禁前置。

### 场景触发条件

| 场景 | 何时激活 |
|------|---------|
| 0.元原意识 | 始终。scope=global 的积木（引擎/门禁域）注入 L0 人格层——Agent 始终知道"我是谁"。 |
| 1.Agent注入 | cognition_loop 执行 on-task 时。按任务关键词从 wiki 取积木→按此顺序拼装→推送给 Agent。 |
| 2.知识积木 | 人类打开 `wiki/知识库/` 下的 .md 文件直接阅读。或 Agent 主动读积木文件时。 |
| 3.Skill文件 | 人类或 Agent 打开 Skill 文件阅读或编辑时。Skill 文件按此顺序组织。 |

## 编排的七字段

| 字段 | 含义 | 例子 |
|------|------|------|
| 组件 | 哪些积木/模块参与 | 做账全流程、安全写入链 |
| 流 | 执行顺序和关系（→串行 ∥并行） | Step1→Step2→...→Step11 |
| 失败 | 每一步坏了怎么办 | config缺失→阻断，cost=manual→⚠️确认 |
| 可变参数 | 什么条件下走不同分支 | taxpayer_type=一般→价税分离 |
| 触发 | 什么事件启动这个流 | plan_review 通过 |
| 设计理由 | 为什么是这个流 | 为什么11步不可颠倒 |
| 已知局限 | 目前覆盖不了什么 | 引擎离线→手动 cognition_loop |

## 四库 + toolkit：积木的物理载体（五层）

积木的存储按五层组织——wiki 下四库 + toolkit 实现层，同层级 `domain/子域/role`，顶文件夹名区分用途。

### 路径结构

```
wiki/
├── 知识库/      声明层——cognition_loop 自动注入
│   └── {domain}/{子域}/{role}/xxx.md   ← 积木文件，全中文
│
├── 参考/        借鉴层——Agent 主动读，不入 injected
│   └── {domain}/{子域}/{role}/xxx.md   ← 同上层级
│
├── 经验/        记忆层——已消化教训
│   └── {domain}/{子域}/{role}/xxx.md   ← N次验证可晋升为知识积木
│
└── 错误/        待处理——还没消化的错误
    └── {domain}/{子域}/{role}/xxx.md   ← 处理后移入经验或知识库

toolkit/         实现层——Python 脚本
└── {domain}/{子域}/tool.py            ← 两层路径，无 role 层
```

### 各库规则

| 库 | 语义 | 规则 |
|------|------|------|
| `wiki/知识库/` | 声明层 | 改知识**只能**改这里。Skill/管线/记忆不存知识。 |
| `wiki/参考/` | 借鉴层 | 只读。不直接引用为执行依据。 |
| `wiki/经验/` | 记忆层 | N 次外部验证后可晋升为知识库积木。N 取决于 confirmed_by 来源累加：晴/KIS 1 次即晋升，auditor 需 ≥3 次，challenger 需 ≥5 次。 |
| `wiki/错误/` | 待处理 | 还没消化的错误。处理后移入经验或知识库。 |
| `toolkit/` | 实现层 | 脚本两层路径 `{domain}/{子域}/`。和 wiki 的 domain/子域 对齐，但无 role 层——Python 不需要按信息形态分目录。 |

四库共享 domain/子域 命名空间。"门禁"在四库中各有存在：知识库有规则定义，toolkit 有 git_safety.py，参考有安全实践，经验有历史教训。

### 五库形态分类

五库按知识形态分为三类，不是五个平行存储层：

| 库 | 形态 | 特征 | 与验证的关系 |
|----|------|------|------------|
| 知识库 | 积木形态 | 单一主张 + 单一role + confirmed_by达标 | 验证通过的声称——"世界应该是这样的" |
| 经验库 | 积木形态 | 单一主张 + 单一role + 待验证 | 待证声称 |
| 参考库 | 源材料形态 | 论文/Skill含多主张，按role排序 | 理论的文字表达——可拆为积木 |
| 错误库 | 事件形态 | 错误记录，按工具来源分类 | 断裂——"这里出过问题" |
| toolkit | 应用实现层 | Python脚本，两层路径 | 证据——"世界实际是这样的"。积木缺失时可逆向还原知识（--help→步骤积木，LLM分析代码逻辑→规则积木） |

## confirmed_by 信任链

每个积木 frontmatter 可携带 confirmed_by 数组。Agent 执行结果反向验证积木——正确提升权重，偏差标记 dirty。

| confirmed_by | weight | N 次验证晋升 | 行为 |
|-------------|:------:|:---:|------|
| 晴 | 1.0 | 1 次 | 必须执行，不可逆。快速通道——晴确认的积木不经过验证周期，直接进知识库 |
| KIS导入报错 | 0.95 | 1 次 | 铁证 |
| auditor | 0.9 | ≥3 次 | 独立 Agent 审计通过 |
| challenger | 0.8 | ≥5 次 | 探针发现盲区后补验证 |
| system | 0.5 | 不可晋升 | 自动工具生成，降低质疑度但不可作为唯一依据 |

晴是快速通道，不是唯一通道。没有晴确认的积木仍可通过 auditor(3次) 或 challenger(5次) 晋升。晴的作用是让关键知识不被验证周期延误。

**边权重：** confirmed_by 覆盖积木节点权重。积木之间的引用边（`references`）另有 edge_reliability 评分——子Agent 发现「定义积木 A 和规则积木 B 互相矛盾」时，这不是 A 或 B 的问题，是 A→B 这条边不可靠。

**往返验证**：Agent 执行结果反向验证积木——正确则提升 confirmed_by 权重，偏差则标记 dirty 或修正。多 Agent 并行时不互斥——各自写回各自的 confirmed_by 条目，cognition_loop 取最高权重条目。

## scope 分流

| scope | 注入层 | 设定方式 | 说明 |
|-------|--------|---------|------|
| `global` | L0 人格层 | frontmatter 显式声明 | 引擎/门禁域默认。每次注入"我是谁"。 |
| `step` | L1-L8 步骤层 | frontmatter 显式声明 | 进项/销项/银行域默认。任务匹配时注入。 |

默认值按 domain 推断：引擎/门禁/编排 domain → global；会计/公司/GUI domain → step。

> ⚠️ **L 编号歧义：** 此处 L0/L1-L8 是 scope 注入层（积木注入路由）。wiki skill 的 L1-L7 是不同的编号系统（七层知识模型：L1=Agent 行为错误、L2=架构设计……L7=财务层）。两者独立——上下文决定当前使用的编号系统。

## 12 个应用面（用 7 role 解释）

> **用途:** 12 应用面是设计框架——验证 7 role 系统覆盖所有使用场景。当提出架构变更时，逐项对照检查是否遗漏。不用于日常执行——日常任务按 AGENTS.md 铁则和三步协议走。

| # | 应用面 | 积木的角色 | 侧重 role | 为什么 |
|---|------|------|------|------|
| 1 | 知识存储 | 积木 = wiki .md 文件 | 定义 + 映射 | 定义积木形态，映射存哪 |
| 2 | Skill 写作 | 积木排列 = skill 文件 | 编排 + 步骤 | 编排积木排列，步骤写出来 |
| 3 | Agent 认知供给 | 积木推送 = cognition_loop 注入 | 编排 + 步骤 | 编排推送顺序，步骤注入机制 |
| 4 | 拓扑依赖图 | 积木引用边 = 跨层依赖 | 映射 | 积木之间的边 = 跨层引用 |
| 5 | 约束对验证 | 积木声称 vs 代码实际 | 规则 | 验证"声称 vs 实际" |
| 6 | 知识复利闭环 | 积木流转 = 经验→知识→门禁晋升 | 步骤 + 规则 | 步骤=晋升流程，规则=晋升条件 |
| 7 | 学习模式 | 积木标准化 = 遍历→补齐→精炼 | 步骤 + 配置 | 步骤=标准化流程，配置=Pomodoro |
| 8 | 编排引擎 | 编排积木 = 7 字段的可执行流定义 | 编排 + 步骤 | 编排=7字段定义，步骤=管线执行 |
| 9 | scope 分流 | 积木元数据 = global/step 注入路由 | 配置 + 规则 | 配置=元数据路由，规则=注入判断 |
| 10 | 门禁标记 | 积木规则分级 = 🛑/⚠️/📎 双机制并行 | 规则 | 规则的分级就是积木标记 |
| 11 | 文件层级 | 积木地址 = domain/子域/role 三维导航 | 映射 + 定义 | 映射=三维地址，定义=层级结构 |
| 12 | 架构设计 | 积木自我应用 = 用积木描述积木体系 | 定义 + 编排 | 定义=设计决策，编排=系统组合 |

### 应用面扩展：role注释双向锚点

积木和代码通过role注释相互验证。积木文件声明了规则→代码中必须有对应role注释（# [规则] 声明内容）。代码中有role注释→积木库中必须有对应文件。少一个就是断层。

这是12应用面中#5"约束对验证"的工程实现——不是"积木声称vs代码实际"的抽象对比，是逐条声明的精确锚定。

从"积木怎么存"（1/11）到"积木怎么串"（2/3/8）到"积木怎么验"（4/5/10）到"积木怎么进化"（6/7）——全链路用同一套 7 role 描述。

### 编排 role vs 编排引擎（应用面 #8）

编排 role 是一个**信息形态**——描述一条可执行流（用"编排的七字段"模板填入内容）。编排引擎是**运行时**——读取编排积木的内容，按流定义调度组件执行。关系：编排积木是编排引擎的**输入**。编排引擎不产出编排积木——编排积木由人手写或 brick-pipeline 提取。

### Skill 五类 vs 12 应用面

五类 Skill（流程/调用/概念/约束/参考）是对应 role 子集的自然归类——和 12 应用面是正交维度。应用面描述"积木在什么场景下被使用"，五类描述"Skill 文件本身属于哪种结构"。一个流程类 Skill（如 accounting）在应用面 #2（Skill 写作）和 #8（编排引擎）中都有角色。

### 积木自我应用（#12）的落位

"用积木描述积木体系"——这些 meta-积木放在 `wiki/知识库/编排/` 和 `wiki/知识库/方法/` 下。编排 domain 存放"积木怎么串"的编排积木（如 cognition_loop 推送流），方法 domain 存放方法论积木（如迁移方法、架构模式）。不存在叫"积木"的 domain——积木是信息单元形态，不是领域。`积木概念.md` 本体放在参考目录，不放入 wiki 层级。

- 流程类：🛑门禁有内容 → 多组件+流+门禁+降级同时出现（如 accounting）
- 调用类：有步骤+策略，但缺门禁 → 单组件（如 hanako-search）
- 概念类：有步骤+原则，但缺门禁 → 概念阶段流（如 hanaagent-migration）
- 约束类：只有定义+规则+步骤+映射 → 纯约束规范（如 pre-commit-verifier）
- 参考类：只有定义+步骤+配置+映射 → 纯资料（如 hermes-provider-config）

> **两套分类对应：** 此处的五类（skill-authoring 标准）与 wiki skill 模板一一对应——流程类=执行类、调用类=工具类、概念类=方法论类、约束类=规范类、参考类=纯文档类。同一个 Skill 用哪套描述看语境：想拆积木用前者，想选模板用后者。

## 设计哲学：隐式知识 vs 显式积木

LLM 内部本身就是一个隐式的知识系统。训练过程将海量知识压缩进数十亿参数——定义、规则、步骤、例子全部混在一起，以分布式权重存在。NeurIPS 2024 的 Knowledge Circuits 论文证实：特定知识的编码涉及多个 attention head 和 MLP 层协作——没有"一个神经元一条知识"的显式边界。

这意味着三个根本限制：

**1. 不可审计。** 你不知道"进项税额的定义"存在哪 7 个神经元还是 700 个。改了一条规则（税率从 13% 调为 11%），无法确认模型内部是否真的更新了——微调影响所有知识，改这个可能坏那个。

**2. 不可定点修复。** Agent 在某个知识断层处反复犯错——你只能改 prompt 让它"注意"，不能定位到"第 7042 个 MLP 神经元存的是旧规则"然后只修那一个。

**3. 只有置信度，没有质疑度。** LLM 的 next-token 采样只有一个轴——这个 token 有多大可能是对的。没有第二个轴——我有多不确定。面对知识断层，它不会说"我不知道"——因为总有概率最高的 token，它选那个。ICML 2026 的"Express Your Doubts"论文指出：token softmax 概率不应被视为"世界事件概率"——模型内部的置信度和"我有多确定"是两回事。

积木体系是对这三个限制的工程回应。积木是离散的、可审计的、可定点修复的——改税率只需改 `会计/税费/配置/增值税税率.md` 一个文件。confirmed_by 信任链是显式置信度。对抗性验证循环（子 Agent 零上下文审计）是工程化的质疑度——LLM 内部不可能存在的自我质疑，被外化为独立的、可量化的系统组件。

**积木不替代 LLM 的知识——积木和 LLM 各管一层。** LLM 管模式识别（"这条发票像不像进项"），积木管精确规则（"暂估冲回必须在新增进项之前"）。两层各管一层、互相补位。

**积木唯一知识源的另一层含义：让"提供"成为可能。** Agent 获取知识有两种模式：

- **搜索**：Agent 先开始想 → 意识到缺了什么 → 自己去查。问题是"意识到缺了什么"这一步本身依赖 Agent 的自我监控能力——而这个能力也是 next-token 采样的一部分，在断层处同样会失效。断层同时让 Agent 拿错知识，也让 Agent 意识不到自己拿错了。RAG 和向量检索都是搜索——在断层处天然有盲区。

- **提供**：知识在 Agent 思考之前已经在上下文里了。cognition_loop 按任务关键词匹配积木，在 Agent 思考前注入。Agent 不需要"意识到自己缺了什么"——它面前已经有完整的积木拼图，直接从拼好的图中开始推理。

提供的前提是知道提供什么。如果知识分散在 Skill 文件、管线配置、对话记忆三处，cognition_loop 无法确定"应该注入哪些知识"。积木是唯一的——所有知识都在 `wiki/知识库/` 下，按 domain/子域/role 三层索引——所以注入是确定的。**积木体系让提供成为可能，提供让 Agent 不需要搜索。因果链从根上重排。**

**提供与对抗性验证：同一个断层，两个方向。** 提供在前面堵——Agent 还没开始想，积木已在上下文，断层不会出现。对抗性验证在后面补——Agent 写完了文件，子 Agent 发现压缩缺口，经验归档，断层变积木。一个是预防，一个是修复。因果方向不同，闭环是同一个：知识断层被消除。
