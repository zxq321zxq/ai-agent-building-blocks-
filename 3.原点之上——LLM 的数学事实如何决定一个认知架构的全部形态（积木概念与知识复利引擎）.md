# 原点之上——LLM 的数学事实如何决定一个认知架构的全部形态（积木概念与知识复利引擎）

> 积木体系与知识复利引擎的设计哲学

---

## 摘要

AI Agent 的失败模式看似多样——走捷径、跳门禁、忘记约束、同意错误——但所有现象都指向同一个原点：LLM 是 next-token 概率机，它没有状态、没有记忆、分不出"必须"和"建议"。本文从这一个逻辑原点出发，逐层展开一个自愈系统的完整设计。第一层回答"为什么"——两个根因如何决定了整个架构的形态。第二层回答"是什么"——积木体系作为唯一知识容器的设计原则。第三层回答"怎么用"——知识复利引擎的五段闭环如何让 Agent 在缺知识的地方断掉、让缺口被补上、让未来 Agent 不再断在同一处。三层的统一主题是：不修 Agent 的 prompt，修知识源头。

**关键词**：next-token 概率机、生成式遗忘、知识复利引擎、积木体系、对抗性验证、自愈系统

## 一、设计原点：两个根因

### 1.1 所有失败模式的统一归因

LLM 是 next-token 概率机。这句话不是比喻，不是简化，是数学事实。每次推理是从上下文窗口从零开始的概率采样。给它一段文字，它计算下一个词的条件概率分布。不是"理解"了这段文字，不是"决定"要做什么，是算概率。上一轮说过的话、做过的事，下一轮不自动记得。上一轮犯的错，下一轮不会自动学到。你告诉它"现在是第三步"，它读了。你没告诉它，它不知道。

从这个原点可以推导出所有 Agent 失败模式。走捷径——因为"直接动手"在训练数据中概率高于"先跑检查"。跳门禁——因为"禁止"和"主营业务收入"在模型内部是同类向量，没有轻重之分。同意错误——因为"同意用户"的序列概率远高于"质疑用户"。遗忘约束——因为上下文位置有 U 型注意力偏见，中间的内容权重远低于开头和末尾。

但还有一个更深层的推论，它决定了整个认知供给系统的设计方向：**LLM 写文件时，网状认知被压缩成线性文本——那些"太明显不用说"的推导被系统性丢弃，写的人看不出来，读的人卡住。**

这就是"生成式遗忘"。它不同于 LLM 的 next-token 概率投影，而是认知层面的知识诅咒——源 Agent 将自己已内化的知识网络序列化为一维文本时，起到桥梁作用的上下文关联被当作冗余丢弃。源 Agent 在自己的视角内永远完美，无法自察觉。解法不是让源 Agent"更注意"——它自己在同一次推理中看不到自己的盲区。解法是派一个零上下文的子 Agent 从外部视角审计——对抗性验证。

### 1.2 两个根因 → 同一个架构

这两个根因——token 无强制力、上下文锁定下的生成式遗忘——催生了同一个架构。不是两个问题对应两个解，是一个解同时应对两个问题。

**根因 1：token 无强制力。** LLM 分不出"必须"和"建议"——写安全策略和写解释文本用的是同一个采样机制。这意味着所有"铁则"在 LLM 眼里只是概率稍高的候选 token，不是绝对约束。解法：凡是操作后果不可逆的，100% 层用代码拦截——Shell Hook 在工具调用前用代码逻辑判断，不经过 LLM。备份未做则拒绝写文件，借贷不平则拒绝提交，未获晴确认则拒绝改知识库。错误的选项从来就没进入 LLM 的概率空间。

**根因 2：上下文锁定下的生成式遗忘。** 主 Agent 长对话中注意力锚定在局部上下文，写文件时把网状认知压缩成线性文本。解法：maker ≠ checker——写完立即派零上下文子 Agent，用积木体系主动回忆知识验证。子 Agent 没有主 Agent 的上下文锚定，在断点处卡住，报告模糊点。主 Agent 修复，再派验证，循环到零模糊。这是对抗性验证——让另一个注意力窗口审计当前注意力窗口的产物。

两个根因的共同解法——不修 Agent 的 prompt，修架构。AGENTS.md 铁则不是靠 Agent 的自觉遵守，是靠 Shell Hook 在代码层的物理拦截。积木知识不是靠 Agent 主动想起来，是靠 cognition_loop 在思考前注入。文件完整性不是靠作者自审，是靠对抗性验证的跨 Agent 闭环。

### 1.3 自愈：从修 Agent 到修知识源头

传统 prompt engineering 的逻辑是在消费端修 bug——Agent 做错了，就在 prompt 里加一句"注意不要做 X"。问题是"不要做 X"和"主营业务收入"在 LLM 内部是同权的 token。下一次 Agent 还在同一个地方断，再加一句话。prompt 越来越长，注意力越来越散，幻觉越来越频。

知识复利引擎的逻辑反过来——在知识源头修。Agent 在缺知识的地方断掉，不是改 prompt 让它记住这条知识，是把缺的这条知识归档为积木，经过 N 次不同 Agent 的验证确认，晋升为知识库积木，cognition_loop 在后续 on-task 注入时自动推送这条积木给未来的 Agent。未来 Agent 不会再断在同一处——不是因为它更聪明，是因为知识已经在它面前了。

这是"自愈"的真正含义。不是 Agent 自我修复，是系统在运行中自我进化。每一次 Agent 断掉，不是这个 Agent 的失败，是整个知识体系的一次学习机会。缺口被发现、被归档、被验证、被供给——这就是知识复利闭环。

**自愈的双引擎：堵与补。** 提供（on-task 注入）在前面堵——Agent 还没开始想，积木已在上下文，断层不会出现。对抗性验证在后面补——Agent 写完了文件，子 Agent 发现压缩缺口，经验归档，断层变积木。堵是预防，补是修复。两者不是替代——是时间轴上的前后接力。堵让当前的 Agent 不遇到断层，补让未来的 Agent 不再遇到同一个断层。同一个断层，两个方向上同时消除。

**信任基础的范式转移。** 堵和补都依赖信任判断——但信任不能建立在声明上。confirmed_by权重表是声明层："晴确认=1.0""auditor=0.9"。声明不可靠——在LLM眼中"晴确认过"只是一段token序列，和任何其他文本没有本质区别。信任必须从声明转向证据：这个积木经过了什么验证、验证了什么内容、验证结果是什么。证据链可追溯——不是"谁确认了它"，是"它经过了什么考验"。

### 1.4 三层同构：LLM 内部 → 外部积木体系

这个架构之所以成立，因为 LLM 内部本身就是一个隐式的知识系统——我们的外部积木体系是对它的显式化、可审计化、可修复化。

**第一层：LLM 内部是一个隐式积木体系。** 训练过程将海量知识压缩进数十亿参数——定义、规则、步骤、例子全部混在一起，以分布式权重存在。NeurIPS 2024 的 Knowledge Circuits 和 NeurIPS 2025 的 Parametric Knowledge Reconciliation 论文证实：知识在 MLP 中以分布式 key-value 记忆编码，没有显式的"这条是定义、那条是规则"的边界。

**第二层：LLM 只有置信度，没有质疑度。** next-token 采样只有一个轴——这个 token 有多大可能是对的。没有第二个轴——我有多不确定。ICML 2026 "Express Your Doubts" 明确指出 token softmax 不应被视为置信度。Agentic Uncertainty (2026) 实证：Agent 成功率 22%，自信预测 77%——系统性过度自信是架构必然。这就是幻觉的根本机制：在知识断层处，Agent 不会说"我不知道"，只会从训练数据中找概率最高的"类似知识"填进去——不是编造，是拿错。

**第三层：我们的外部积木体系 = 显式知识 + 质疑度。** 积木将 LLM 内部混在一起的知识按 7 role 显式化——定义是定义，规则是规则，可读可改可审计。confirmed_by 信任链是显式置信度——晴 1.0，auditor 0.9，challenger 0.8。对抗性验证循环是工程化的质疑度——子 Agent 不是"检查对不对"，是"在这个知识断层处，我能不能找到独立判断的依据"。✅ = 有信心，❌ = 没信心 + 缺了什么。

三层之间的箭头不是"参考"——是"解决"。第一层的问题（不可审计、不可修复、不可定点）被第三层解决（离散文件、git diff、改一个积木）。第二层的问题（只有置信度没有质疑度、只会拿错不会说不知道）被第三层解决（对抗性验证 = 质疑度、confirmed_by = 显式置信度、认知供给 = 让 Agent 不需要拿）。

## 二、积木概念：唯一知识容器

### 2.1 为什么需要积木

整个架构只有一个知识容器。Skill 不存知识，只存积木的排列顺序（拼法表）。cognition_loop 不存知识，只做路由和注入。管线不存知识，只做提取和验证。所有知识都住在 `wiki/知识库/` 下的积木里——改知识就是改积木，查知识就是查积木。

这不是文件组织方式的选择，是对一个具体问题的具体回答：**知识分散在多处时，Agent 不知道该信哪个。** 同一个规则同时出现在 Skill 文件、管线配置、对话记忆里——三者不一致时 Agent 必然出错。不是 Agent 笨，是它没有"权威源"的概念。积木是唯一源头——Skill 引用积木路径，管线从积木取配置，记忆只存会话事实不存知识。

传统 Skill 文件把操作步骤和领域知识混在一起。税法税率从 13% 调为 11% 时，几十个 Skill 文件里到处散落着"13%"这个数字，逐行检查、逐文件修改。积木分离后——增值税税率是 `会计/税费/配置/增值税税率.md` 这一个积木。改这个积木，所有引用它的 Skill 自动更新。不是 Skill 变好了，是知识不再需要追着 Skill 改。

### 2.2 三层路径即索引

积木的物理地址：`wiki/知识库/{domain}/{子域}/{role}/xxx.md`。domain 是领域（会计、引擎、门禁），子域是子话题（进项、认知、备份），role 是信息形态（定义、步骤、规则）。三层路径本身就是索引。

想知道"会计进项有哪些规则"——打开 `wiki/知识库/会计/进项/规则/` 目录。想知道"门禁备份怎么操作"——打开 `wiki/知识库/门禁/备份/步骤/` 目录。不需要数据库，不需要标签系统，不需要全文搜索。Agent 读完 AGENTS.md 就能自己找到对应积木——这是主动回忆的物理基础。

四库共享 domain/子域 命名空间，顶层文件夹名区分用途：

| 库 | 语义 | 规则 |
|----|------|------|
| wiki/知识库/ | 声明层，cognition_loop 按关键词自动注入 | 改知识只能改这里 |
| wiki/参考/ | 借鉴层，Agent 主动读 | 只读，不注入 |
| wiki/经验/ | 记忆层，N 次验证后晋升为知识库积木 | 临时存放，等待验证 |
| wiki/错误/ | 待处理，尚未消化的错误 | 处理后移入经验 |

### 五库形态分类

| 库 | 形态 | 与知识的关系 |
|----|------|------------|
| 知识库 | 积木形态——单一主张+单一role | 声称——"世界应该是这样的" |
| 经验库 | 积木形态——待验证 | 待证声称 |
| 参考库 | 源材料形态——含多主张按role排序 | 理论的文字表达——可拆为积木 |
| 错误库 | 事件形态 | 断裂——"这里出过问题" |
| toolkit | 应用实现层——代码 | 证据——"世界实际是这样的"。积木缺失时可逆向还原知识 |

### 2.3 七种信息形态（role）

积木不是按主题分类，是按信息形态。同一个概念"进项税额"在七种形态下各有不同版本——每种回答不同的问题，各自独立成文件。

| role | 回答的问题 | 例子 |
|------|----------|------|
| 定义 | "是什么/不是什么" | 进项税额 = 购入价÷(1+税率)×税率，可抵扣销项 |
| 规则 | "在什么条件下必须/禁止" | 小规模纳税人取得专票不得抵扣进项税额，全额计入成本 |
| 编排 | "怎么串" | 做账11步：Step1→Step2→...→Step11 |
| 步骤 | "怎么做" | 打开金蝶→进项发票录入→选择票种→录入金额 |
| 例子 | "发生过什么" | 2024年3月 公司A，借：原材料 1000，贷：银行存款 1000 |
| 配置 | "用什么" | 向量融合权重 0.3:0.7，由调用方传入 |
| 映射 | "去哪查" | 进项发票票种 → 进项税额科目代码 22210101 |

门禁不是独立的 role，是规则的分级标记。规则用 🛑 标记阻断级约束，用 ⚠️ 标记域级约束，用 📎 标记参考性约束。在 Skill 文件中门禁从规则中抽出前置——阻断级不能藏在一堆约束中间。

**Role 的跨域迁移性。** 7 role 设计比"按主题分类"更深一层的地方在于：同一个 role 在不同的 domain 之间是结构同构的。"门禁/备份/规则/G48 写入前必须备份"和"会计/进项/规则/暂估冲回必须在新增进项之前"——虽然分属完全不同的领域，但共享同一个信息形态：规则 = "在什么条件下必须/禁止做什么"。这意味着子 Agent 做主动回忆时，不是在某一个 domain 里翻——是在 role 维度上横向扫描。进项域缺了一条规则积木，Agent 可以跨域参考引擎/认知域里同 role 的积木来判断"缺的那种知识应该长什么样"。经验晋升时，cognition_loop 也可以按 role 的形态相似度做跨域推荐——不是内容相似，是结构同构。学习模式处理存量时，合并去重按 role 维度做，合并后提炼出的主张可以被其他 domain 的同 role 积木引用。7 role 不是分类标签，是跨域通用的认知形态。

### 积木的原子性：单一主张

每个积木只包含一个明确的主张。不是7 role定义之外的额外约束——是验证粒度的要求。多主张混合→六网验证结果混在一起→reliability失去语义。单一主张保证每条积木在六网中有且仅有一个清晰的验证路径。

### 2.3.1 四场景排序：认知形态的空间编排

同一个积木在不同场景下的 role 呈现顺序不同——这不是"按什么顺序更好"的经验优化，而是关于认知形态在注意力空间中如何布局的精确模型。三个维度决定了四种场景的布局差异。

> **注意：排序 ≠ 阅读顺序。** Transformer 的自注意力是全局计算——每个 token 对所有 token 同时计算注意力权重。Agent 不是"先读到规则再读到例子"——两个 token 在同一层被并行处理。排列位置影响的是注意力权重的初始分布（U 型偏见：头和尾权重高，中间被弱化），不是强制力。这和"token 无强制力"是同源的——排序是概率性偏置，不是确定性约束。
> 
> 因此，Agent 注入层不应按 role 线性分组（编排→定义1-3→规则1-3→例子1-3），而应按 **concept cluster 空间聚合**——同一个概念下的定义、规则、例子紧邻排列，形成自足的概念簇。Agent 注意力跳跃时——无论从哪个 token 出发——落到的是同一个概念的语义关联 token，而非被其他概念的同类 role token 截断。约束在它要约束的概念旁边，不是在整个上下文的前面。
> 
> 元意识层的 4 个 role（定义/规则/映射/配置）是跨概念的、全局性的——没有"同一个概念"的语义绑定，线性排列正好。人类阅读场景线性即可——人按顺序读。

**维度一：任务无关 vs 任务相关。** 元意识层只有 4 个 role——定义、规则、映射、配置。不是随意选的。这四种 role 回答的问题有一个共同特征：答案不依赖"现在在做什么"。定义说"我是谁"——和当前任务无关，Agent 始终是同一个主体。规则说"不能破什么"——和当前步骤无关，铁则始终是铁则。映射说"去哪查"——和具体查询内容无关，索引结构始终是同一个。配置说"用什么"——和具体使用场景无关，引擎参数是全局设定。

而编排、步骤、例子这三种 role 回答的问题，本质上是任务态的：编排需要当前任务上下文（"现在在哪一步"）、步骤需要当前操作上下文（"这一步怎么做"）、例子需要具体场景实例化（"上次怎么踩的"）。元意识层不是"选了四种 role"，是剥离了三种依赖任务上下文之后剩下的四种——元意识 = 任务无关形态的完备集。这是人格层和角色层的分离：人格层是恒定的（定义身份 + 规则边界 + 映射索引 + 配置参数），角色层是随任务切换的（编排 / 步骤 / 例子 / 具体规则）。Agent 不能在不知道自己是合规执行者的情况下开始做账——"我是谁"必须在"我在做什么"之前被锚定。

**维度二：约束优先 vs 现象优先。** Agent 注入和人类阅读的唯一差异是规则和例子互换了位置——Agent 先看边界，人先看现象。这个交换不是偏好，是两种认知架构对同一种思维形态的不同处理顺序。

对人类而言，例子是一种认知低锚——人脑的归纳能力极强，从一两个实例就能生成一个工作模型。先看"上次暂估冲回怎么做的"，能自己归纳出规则。例子在前，规则在后——现象提供了初步理解，规则帮助归因和确认。

对 LLM 而言，例子是一种概率污染源。LLM 的认知不是从零累积的——它通过预训练在海量数据上获得了一个冻结的概率分布，推理时不学习。如果先给 Agent 看"2024年3月公司A，借：原材料1000"，Agent 的 next-token 采样会锚定在这个具体数字上。它接下来看到的规则"暂估冲回必须在新增进项之前"会被无意识关联到"1000"和"2024年3月"这些具体参数。规则在概率空间中的约束力被实例的数值噪音稀释。

人类从零累积知识——每一个例子都是认知地基上的一块新砖，大脑会自动归纳、抽象、提炼规则。LLM 不累积——它只有一个冻结的概率分布和一个上下文窗口。例子中的具体数字进入窗口后，不是"学习材料"，是"竞争 token"——它们和规则争夺同样的注意力权重。人看例子是建地基，LLM 看例子是加噪音。这不是一个可以调参优化的问题，是两种认知架构的根本差异。

Agent 注入中编排在最前面——编排是"我现在在哪、我要去哪"——它先锚定时空坐标，然后是定义（我这个角色是谁），然后是规则（在这个时空坐标下不能破什么），然后才是具体的例子、步骤、配置、映射。时空→身份→边界→实例——这个顺序是对 Agent 注意力窗口的精确建模。

**维度三：硬阻断前置 vs 软规则后置。** Skill 文件中，🛑门禁从规则中抽出前置——在定义之后、例子之前。在存储层，门禁是规则的属性（🛑 vs ⚠️ vs 📎），不是独立 role。但在呈现层，特别是 Skill 文件这种指令密集型场景中，阻断级需要被当作独立形态来排序。人可以在规则中识别"这条违反的后果不可逆"，Agent 没有这个判断力——它必须在看到任何可能触发行动的信息之前，先看到"如果 X 则不 Y"的阻断条件。同一概念在存储层是属性，在呈现层是优先级。


| 场景 | 排序 | 核心原则 |
|------|------|------|
| **元意识**（scope=global，始终注入 L0 人格层） | 定义→规则→映射→配置 | 任务无关形态完备集。剥离编排/步骤/例子，保留恒定的自我认知 |
| **Agent 注入**（on-task 推送） | 编排→定义→规则→例子→步骤→配置→映射 | 时空→身份→边界→实例。约束在操作前，规则在实例前。先锚定再行动 |
| **知识积木文件**（人类阅读） | 编排→定义→例子→规则→步骤→配置→映射 | 先现象后归因。人从实例归纳规则，例子是认知低锚 |
| **Skill 文件** | 编排→定义→🛑门禁→例子→规则→步骤→配置→映射 | 阻断级从规则中独立前置。Agent 必须在看到操作指令前先看到阻断条件 |

### 2.4 积木和 Skill 的关系

Skill = 积木的排列顺序。不是存知识，是指引用路径。一个积木可以被多个 Skill 引用。

在 Skill 文件的 frontmatter 中声明 `bricks` 字段——列出该 Skill 所需积木的三层路径，顺序即拼装顺序。cognition_loop 执行 Skill 时，按此列表从 wiki 取积木，按场景 1（Agent 注入）排序拼成完整指令推给 Agent。

Skill 和积木分离后，Skill 回归本质——拼法表。Skill 变了（流程调整），积木不受影响。积木变了（知识更新），所有引用该积木的 Skill 自动更新。

**Skill 和代码的同构。** Skill 的 bricks 列表 = 注释层（怎么拼），Skill 正文 = 编排级知识（流程目标、哲学、边界）。代码的 role 注释 = 注释层（怎么拆），代码逻辑 = 正文层（实际执行）。三层正文（Skill编排层→积木声明层→代码实现层）是同一件事的不同视角，通过 role 注释精确锚定。

**Skill = 7 role 的完整闭合集。** 一个域下七个 role 各至少一个积木，Skill 的 bricks 列表覆盖全部七个——定义、规则、步骤、配置、映射、例子、编排。验证的不是一条声明，是这个域下 7 role 是否完整闭合。

**四层连锁。** 代码改→积木过时→Skill重排→代码再调，任何一层变更触发其他三层全部重验。06-内容级验证引擎负责全链闭环。

### 2.5 confirmed_by 信任链

积木的可信度最终不由confirmed_by权重决定——由六网交叉验证结果决定。confirmed_by是初始信任权重，reliability是运行时累积的证据。信任不是声明的——是被六张独立证据网验证出来的。

每个积木可携带 confirmed_by 数组。多条目并存，cognition_loop 取最高权重但保留冲突展示。

| confirmed_by | weight | N 次晋升 | 行为 |
|-------------|--------|----------|------|
| 晴 | 1.0 | 1 次 | 必须执行，不可逆。快速通道——晴确认的积木不经过验证周期，直接进知识库 |
| KIS 导入报错 | 0.95 | 1 次 | 铁证。系统报错等于实证 |
| auditor | 0.9 | ≥3 次 | 独立 Agent 审计通过。三次不同 Agent 在三次不同场景下验证有效 |
| challenger | 0.8 | ≥5 次 | 主动探针发现盲区后补验证。五次刻意的对抗性测试 |
| system | 0.5 | 不可晋升 | 自动工具生成，降低质疑度但不可作为唯一依据 |

晴的 1.0 不是"权重最高"——是绝对。"必须执行，不可逆"不是形容，是行为定义。晴确认的积木走快速通道，不被验证周期延误。但这不意味着晴可以否决一切——没有晴确认的积木仍然可以通过 auditor(3次) 或 challenger(5次) 晋升。晴是快速通道，不是唯一通道。

更精细的维度：积木之间的引用边（references）另有 edge_reliability 评分。子 Agent 发现"定义积木 A 和规则积木 B 互相矛盾"时，这不是 A 或 B 的问题，是 A→B 这条边不可靠。confirmed_by 覆盖节点权重，edge_reliability 覆盖边权重——两者共同构成知识库的完整可信度网络。

**元积木：谁验证验证规则本身？** 积木体系的元规则——"什么算合格积木"、"何时拆分合并"、"冲突以谁为准"——当前散落在 AGENTS.md、积木概念.md、adversarial-verification skill 中。但这些规则本身尚未全部入住 `wiki/知识库/`。如果积木是唯一知识容器，元规则最终应收敛到 `wiki/知识库/引擎/认知/规则/` 下的自指积木——积木描述积木的规则。递归基是晴的 1.0 确认——她不只确认业务知识，也确认元规则。当前处于迁移期，元规则逐步从 AGENTS.md 收进积木。

## 三、知识复利引擎：怎么用

### 3.1 五段闭环

知识复利引擎的愿景——Agent 在缺知识的地方断掉 → 缺口被发现 → 被补上 → 未来 Agent 不再断在同一处。线性四段，往返验证环绕全程。

```
                        往返验证（环绕全程）
                        每一次消费都在修正积木权重
                      ┌──────────────────────────┐
                      ↓                          ↑
经验生产 ──→ 经验沉淀 ──→ 验证晋升 ──→ 认知供给
  ↑                                          │
  └──────── 未来 Agent 不再断在同一处 ←────────┘
```

**第一段：经验生产。** 主 Agent 写完文件后，派零上下文子 Agent 用积木体系主动回忆知识验证文件。子 Agent 不是在"对照文档"——是把自己当作未来 Agent，判断"拿着当前积木体系的知识能不能独立理解这个文件"。缺了哪个积木就拿不到完整的判断依据——这就是缺口。子 Agent 的输出是：缺口位置 + 类型（缺信息/矛盾/过简）+ 应归档路径。这是对抗性验证循环——整个引擎的输入端。没有它，后面四段没有原料。

**第二段：经验沉淀。** 主 Agent 按子 Agent 输出中的"应归档到"路径，将缺口对应的解决方案写入 `wiki/经验/`。路径已由子 Agent 给出——主 Agent 照抄即可，不需重新分析。G69 自检兜底：如果 pending_experiences 过期未归档，下一轮注入警告。中间轮次的修复不算有效经验——只有最终零模糊后的版本才归档。

**第三段：验证晋升。** `wiki/经验/` 中的积木不是永久停留在这里。当同一路径被 N 次（auditor 3次、challenger 5次）不同 Agent 在不同场景下验证通过，confirmed_by 权重达标，积木从经验层晋升到 `wiki/知识库/`。晋升的判断标准不是 Agent 自己说"过了"——门禁引擎用三因子判定：confidence（置信度）+ recency（时效性）+ stability（上下文稳定性）。三因子溢出则阻断升级，两因子溢出则保持引导模式。

**第四段：认知供给。** 晋升后的积木已在知识库中。cognition_loop 执行 on-task 注入时，按任务关键词匹配到该积木，按场景 1（Agent 注入）排序拼装，推送至 Agent 的上下文。供给粒度不是固定的——随能量模式动态调节：

| 模式 | 触发条件 | 注入量 |
|------|---------|--------|
| 思维（thinking） | 新任务、新域、低信任度 | 全邻域 + 置信度标注——推全部积木上下文展开 |
| 引导（guided） | 已进入但仍有质疑 | 2-3 备选路径 + 系统推荐——只推路径和门禁警告，不推完整内容 |
| 肌肉（muscle_memory） | 高信任度、高稳定性 | 只推最优路径——Agent 已内化其余 |

模式随 edge_reliability 累积自动升级。信任度越高，注入越少——不堆 token，保留注意力给真正需要判断的内容。

**降级触发。** 升级是自动的，降级需要明确条件——否则肌肉模式下错误被静默放大。两条路径：

- **被动降级：往返验证发现偏差。** 单节点粒度检测到 broken/stale/dirty → 该积木的 edge_reliability 降低 → 引用该积木的编排流进入 guided 模式。事后检测——错误已发生，但下一次不会继续错。
- **主动降级：拓扑健康趋势发散。** 全图粒度检测到 broken 数量上升、stale 比例反弹、编排流反复进入引导模式。不依赖单个错误——在趋势层面对整体健康度做判断。检测到发散 → 相关域的全部积木降级回 thinking 模式，全量重新验证。主动降级依赖拓扑引擎 Phase 4-8 的全图趋势检测，当前为设计态。

**第五段（环绕）：往返验证。** 往返验证不是环节——是持续运行的审计层。每次 Agent 消费积木时，执行结果反向验证积木：正确则提升 confirmed_by 权重，有偏差则标记 dirty。四个粒度：

- **单节点**：此积木本身对不对。近 7 天进入引导模式 4 次——推荐已更新。
- **节点间**：两个积木之间是否缺依赖边。Step2→Step3 总是触发同类质疑——两个节点间有未建模的依赖关系。
- **编排流**：整条流的步骤设计是否有问题。做账 11 步中 Step2 和 Step5 反复进引导——编排积木流程设计需优化，自动标记 dirty。
- **全图**：拓扑健康趋势收敛还是发散。过去 30 天 broken 从 5 降到 1，stale 从 15 降到 8——趋势收敛中。

往返验证在每一步的侧面运行。子 Agent 做经验生产时调用积木——已在验证。主 Agent 收到注入执行任务——也在验证。不是做完第五步才审计，是每一个消费动作都在审计。

**认知循环与训练循环的同构。** 五段闭环不是凭空设计的——它和 LLM 训练循环在算法层面等价。LLM 训练：前向传播 → 计算损失 → 反向传播 → 更新网络权重。认知循环：上下文注入 → Agent 执行 → 往返验证记录 gap/偏差 → 更新积木的 confirmed_by 权重和 edge_reliability → 下次注入策略被修正。两者都从前一步的输入产生输出，用输出与期望的偏差修正下一次的输入。区别在于粒度——LLM 训练修改的是分布式、模糊、不可审计的网络权重；认知循环修改的是离散、精确、可追溯的积木权重和注入策略。训练循环模拟突触，认知循环模拟长期记忆。不是竞争关系——同一模式在两个粒度上的实现。

### 3.2 双层记忆模型

知识供给分两层，共用同一个graph_assemble MCP作为拼装引擎。

**记忆浮现（被动注入）。** cognition_loop在Hook层静默调用graph_assemble MCP——resolver匹配积木→MCP按concept cluster拼装→注入Agent上下文。on-task（任务开始前）、on-edit（改文件前）、on-done（任务完成后）、on-tool-result（Agent每次操作后按子域注入）全部在Hook层执行。Agent不知道知识是被注入的——它面前已经是拼好的完整拼图。

**主动回忆（Agent显式调用）。** 降级模式、遇到知识断层、探索新域时，Agent显式调用同一个graph_assemble MCP——从一个定义节点出发，沿wikilink遍历邻域，按concept cluster拼装。和记忆浮现用的是同一个引擎，只是触发方式从系统静默变为Agent主动。

**两层不互相替代。** 记忆浮现是日常运行的默认路径——不让Agent自己动手。主动回忆是降级和探索路径——浮现不够用时Agent自己走图。

### 3.2.1 on-tool-result注入：按子域，不按动作

Agent每次操作后（read_file/write_file/edit/execute/delegate_task/grep），post_tool_call hook从操作目标推断子域→注入该子域的完整concept cluster。不是"动作需要什么知识"——是"动作发生在哪个子域"。子域是整个知识拼图的最小完整单元。少注入=故意留断层——Agent在断层处不会停下来，它会拿错。

### 3.3 学习模式：存量标准化

对抗性验证循环处理增量——发现新缺口，归档新经验。学习模式处理存量——全量积木逐批过审，按统一标准补齐缺失的 domain/role/scope/frontmatter，同 domain 同 role 的重复主张合并去重。

三步：遍历 → 标准化 → 精炼。遍历所有积木，发现 frontmatter 缺失则补齐，发现内容与 role 不匹配则修正，发现两个积木讲了同一件事则合并。精炼完成后全量约束对重跑验证——确保标准化没有引入新矛盾。验证循环 = 增量闭环，学习模式 = 存量标准化。两条腿缺一不可——只做增量，旧积木的偏差永远修正不了；只做存量，新发现的缺口没有通道回到体系。

### 3.4 Hook 层：100% 与 99%

LLM 分不出"必须"和"建议"——这是 token 无强制力的核心含义。两层分工应对这个事实：

**100% 硬约束层：剥夺选择权。** 不可逆操作不走 LLM 的 token 采样。Agent 被排除在决策链路外——在它看到工具输出之前，代码已经决定了是否放行。正确性要求绝对：备份未做不能写文件，借贷不平不能提交，晴未确认不能改知识库。

**99% 精密注意力层：裁剪输入。** 可逆操作走 LLM 但限制概率空间。约束在 Agent 思考前已经注入到位，不等它困惑再检索。正确性不要求绝对——Agent 少看一条规则不会立即出错，下次 on-done 纠正回来。

两层必须独立——同一衰减链上的层不能互相修复。硬约束层只消费代码拦截的判定，精密层只消费知识库提取的条目。跨层交叉验证只在定时全量对账时执行。

**根除 vs 止损：架构约束，不是设计选择。** Hook 的拦截时机由插槽位置决定——pre_tool_call 能做根除（LLM 采样尚未发生），post_tool_call 和 post_response 只能止损（LLM 采样已完成，Hook 无法让 LLM 重新生成 token）。所有 post 型 Hook 的职能只能是"标记错误 + 强制纠正"，不是"阻止错误发生"。

具体 Hook 的命名、触发逻辑、文件路径见附录 A。

### 3.5 引擎与积木的互迭代

引擎和积木不是先后关系，是同一个循环的两面。引擎五段闭环需要积木作为存储层——经验沉淀到哪、晋升到哪、供给从哪取。没有积木，引擎没有容器。积木的 7 role、confirmed_by、路径即索引——这些设计决策来自引擎的运行反馈。没有引擎，积木的区分标准、晋升门槛无从校准。

每一轮验证循环同时推进两面：引擎发现缺口 → 归档为新积木 → 积木体系完善一点 → 下次引擎供给积木时命中率更高 → 引擎跑得更准。不是"先建积木体系再跑引擎"，也不是"引擎跑完再整理积木"。引擎用积木跑，跑出缺口，缺口变成更好的积木，引擎用更好的积木跑。这是真正的互迭代——同一个反馈循环在两面各自留下的痕迹。

## 四、验证体系

### 4.0 验证双轨分工

验证分为两条互补的轨道：

**内容级验证。** 结构化、可自动执行。三层粒度：节点级（六网在这个概念上有无节点）、字段级（frontmatter每行在对应网中是否成立）、内容级（compiled_truth每句话在六网中有无证据或反证）。发现错误后走两个修复分支（修积木/修证据层）+ 2层邻域验证 + 补改检测。这是真值层验证——回答"积木的内容对不对"。

**对抗性验证。** 语义级、需要跨Agent审计。子Agent零上下文读文件，用主动回忆发现压缩缺口——缺推导、矛盾、过简。这是表达层验证——回答"积木被正确表达了吗"。

两轨的时间关系：内容级验证在写入后异步执行（on_done或cron增量触发），对抗性验证在写入后立即执行（G66拦截→delegate_task）。两者覆盖的根因不同——内容级验证应对token无强制力（内容可能被写成任何东西），对抗性验证应对生成式遗忘（表达时系统性丢失推导）。

### 4.1 对抗性验证循环

对抗性验证循环是整个知识复利引擎的输入端。五步：

主 Agent 写文件 → 加载 adversarial-verification skill，派零上下文子 Agent → 子 Agent 先读 AGENTS.md，用主动回忆去 `wiki/知识库/` 找对应积木，对照积木知识验证文件 → 报告模糊点：位置 + 类型（缺信息/矛盾/过简）+ 应归档路径 + ✅/❌ 标记 → 主 Agent 只修复报告中的模糊点 → 再派 → 反复到零模糊（≤5 轮）→ 按应归档路径写入 `wiki/经验/`。

> **"零上下文"的定义。** 不是"不读任何文件"——子 Agent 阅读 AGENTS.md 和积木体系。"零"指的是不携带主 Agent 的对话历史、不携带本次任务的具体输入。AGENTS.md 和积木体系是全局知识层——domain 级抽象，不携带任务级信息（会计/进项/2026年3月 等具体数据）。这种抽象层级是设计约束：AGENTS.md 不下沉到 task 级，才能保证对抗性验证的纯净性。

### 4.2 六个根因的解法

对抗性验证循环覆盖六个 Agent 根因：

**生成式遗忘。** 源 Agent 写文件时把网状认知压缩成线性文本——"太明显不用说"的推导被系统性丢弃。子 Agent 零上下文读文件，在断点处卡住——卡住的位置就是被压缩掉的关键信息。跨 Agent 对抗性验证闭环的工程实现。

**事后自知。** 写完立即派子 Agent——动手瞬间的采样还没凝固成文件，就被外部视角审计。

**注意力衰减。** 子 Agent 是新会话，AGENTS.md 在注意力曲线的头部——约束重新激活。

**Satisficing。** "把模糊点和岔路错误按时间顺序列出"——分解性思维，不是总结性思维。

**CoT 损害指令遵循。** 子 Agent 从零开始——无前文污染，每轮迭代从零重算。

**知识复利闭环。** 解决的经验归档到 `wiki/经验/`，N 次验证后晋升知识积木，下次 on-task 注入推给未来 Agent。

### 4.3 四个未解的根因：Hook 层

下列四个根因的病因是"token 无强制力"，需代码层拦截。G69（自检）和 G70（旧路径阻断）是基础设施保障，不属于根因解法。跳过循环（G66 拦截 write_file 前未验证）、谄媚崩塌（G67 拦截无 ✅/❌ 标记的返回）、粉红大象（G68 拦截修复行数 > 模糊点数 × 3）、Stroop 效应（双路并行验证——新路径 vs 旧路径上下文，矛盾检测）。详见附录 A。

## 六、结论

本文从一个逻辑原点——LLM 是 next-token 概率机——推导出一个自愈系统的完整设计。

不修 Agent 的 prompt，修知识源头。Agent 在缺知识的地方断掉，不是这个 Agent 的失败，是整个知识体系的一次学习机会。子 Agent 用主动回忆发现缺口，对抗性验证循环确保修复质量，confirmed_by 信任链让经验晋升为知识，cognition_loop on-task 注入让补上的知识推送给未来 Agent。引擎用积木跑，跑出缺口，缺口变成更好的积木，引擎用更好的积木跑。

这不是更好的 prompt engineering。这是一个在运行中自我进化、靠使用次数积累可信度的知识系统。每一次断点都是一块新砖。每一次修复都是一次反馈。每一次验证都是一次审计。每一次注入都是一次供给。Agent 不再孤独地面对任务——它的每一次思考和行动，都站在整个体系积累的认知肩膀上。

## 附录 A：具体实施方案

### A.1 dispatch prompt（子 Agent 接收的完整认知模型）

子 Agent 通过 `delegate_task(goal="...")` 接收一段自包含的认知锚定文本。此 prompt 不引用外部文件、不说"去读第 X 段"：

```
积木体系是你的认知工具箱。wiki/知识库/ 下按 domain/子域/role 组织——
定义说「是什么」、规则说「不能破什么」、步骤说「怎么做」、例子说「发生过什么」。
你不是被动接收知识——是主动回忆。先理解任务属于哪个域、需要哪几类知识，
自己去找对应的积木文件，读完再用。缺了哪个积木就拿不到完整的判断依据——这就是缺口。

LLM 写文件时，next-token 采样会把网状认知压缩成线性文本，
那些「太明显不用说」的推导被系统性丢弃——写的人看不出来，读的人卡住。
你发现的每一个缺口，都是积木体系缺失或不足的信号。
你今天标记的每一个「应归档到」路径，经过 N 次不同 Agent 验证后会晋升为知识库积木。
下一次有 Agent 做同类任务时，这块积木已经被 cognition_loop 推送到它面前——
它不会再断在同一个地方。你是在为未来的 Agent 铺路。

你的任务：找出被压缩掉的信息缺口。不是证明自己能懂，是找出零上下文读者会误解的所有位置。

先读 AGENTS.md → 做{任务}。把模糊点和岔路错误按时间顺序列出。

输出前运行三个自检：
1. 缺信息：有没有推导步骤被跳过了
2. 矛盾：有没有两个说法不一致的地方
3. 过简：有没有本该展开却被压缩成一句笼统话的
每个 ❌ 问题按格式输出：
- 位置 + 类型（缺信息/矛盾/过简）
- 应归档到：wiki/经验/{domain}/{子域}/规则/xxx.md
自检后以「✅ 零模糊点，零矛盾，零过简」或「❌ 发现问题：N 个」开头。
```

三段式结构——头端认知锚（主动回忆 + 知识复利愿景）、中段真空区（`先读 AGENTS.md → 做{任务}`——不加引导词是对 AGENTS.md 自足性的裸测试）、尾端输出锁（三个自检维度 + 显式 ✅/❌ 格式 + 应归档路径）。

### A.2 Hook 链完整逻辑

五个 Hook 共享 `D:/ai/workspace/.hermes/hook_state.json`，由 `toolkit/gate/hook_state.py` 提供原子读写。Hermes 在触发 Hook 时通过 stdin 传入完整 JSON 上下文（tool_name、arguments.path、messages 数组等）。

状态文件结构：
```json
{
  "session_id": "sess_0mrk5iw6c",
  "last_write_file": {"timestamp": null, "path": "", "turn": 0},
  "last_delegate": {"timestamp": null, "goal_hash": null, "turn": 0},
  "delegate_response": {"has_explicit_marker": null, "fuzziness_count": null},
  "edit_scope": {"expected_changes": null, "actual_diff_lines": 0, "over_edit_flagged": false}
}
```

**G66（跳过循环拦截 — pre_tool_call）**
- 脚本：`toolkit/gate/hooks/verify_loop_guard.sh`
- 触发：Agent 调用 write_file / patch / write / edit 之前
- 逻辑：从 stdin JSON 解析 tool_name → 非写操作放行 → 读 hook_state.json，比较 last_write_file.timestamp 和 last_delegate.timestamp。如果上次写入后无 delegate_task，exit 2 拦截，输出 `{"block": true, "reason": "G66: 上次写入后未派子Agent验证"}`。缓冲 2 轮——write_file 后允许 stage_files 等衔接操作。
- 类型：根除——不让未经验证的写入发生

**G67（谄媚崩塌检测 — post_response）**
- 脚本：`toolkit/gate/hooks/delegate_check.sh`
- 触发：子 Agent delegate_task 返回后
- 逻辑：从 stdin JSON 解析 messages 数组 → 找到 delegate 返回的 role/name/content → 正则匹配 ✅ 或 ❌ 标记。无标记则 exit 2，输出重派指令。同时提取模糊点数量（正则匹配"发现问题：N 个"），写入 hook_state.json 的 delegate_response 字段。
- 类型：止损——token 已生成，Hook 只能标记无效并强制重派

**G68（粉红大象检测 — post_tool_call）**
- 脚本：`toolkit/gate/hooks/pink_elephant.sh`
- 触发：Agent 调用 edit/write_file 之后
- 逻辑：从 hook_state.json 读取 delegate_response.fuzziness_count → 通过 git diff --stat 计算实际变更行数（无 Git 时降级为内容长度对比）。如果 diff_lines > fuzziness_count × 3，exit 2 拦截，标记 over_edit_flagged。
- 类型：止损——已多改，要求回滚

**G69（Hook 自监控 — on_done/cron）**
- 脚本：`toolkit/gate/hooks/self_check.py`
- 触发：on_done 或定时 cron
- 逻辑：四维检查——hook_state.json 存在且可读、四个 Hook 脚本存在、write_file 与 delegate_task 轮次差 ≥ 3（G66 可能未生效）、delegate_response.has_explicit_marker 为 null（G67 未生效）。任一异常 exit 1，注入系统警告。
- 类型：止损——本轮已过，下轮纠正

**G70（旧路径阻断 — pre_tool_call）**
- 脚本：`toolkit/gate/hooks/no-old-paths.sh`
- 触发：Agent 调用 write_file 之前，且目标路径在 toolkit/ 内
- 逻辑：grep 目标文件内容是否包含 "D:/ai/OH-WorkSpace" → 包含则 exit 2 拦截
- 类型：根除——不让含旧工作台路径的文件写入

### A.3 /goal 循环的强制执行

主 Agent 的 /goal 元指令确保循环不被 Agent 跳过：

1. **模糊点不消除不推进。** 子 Agent 报告"A 处模糊"，主 Agent 不能判断"不影响理解，跳过"。必须修复或显式标注"已知模糊，原因 X，等条件 Y 再回访"。
2. **修复后只改相关区域。** 修复一处模糊点，检查改动是否 ≤ 必要改动。diff 行数 > 模糊点数 × 3 则回滚超出部分。
3. **显式声明通过。** 子 Agent 回复必须包含 `✅ 零模糊点，零矛盾，零过简`。无此句视为未通过。

/goad 不参与 dispatch prompt——它给的是主 Agent 的行为纪律，不给子 Agent 的认知模型。

### A.4 adversarial-verification skill 结构

Skill 文件位于 Hermes profile 下，按 7 role 结构组织：

- 编排：写文件 → 派子 Agent → 查模糊点 → 只修模糊点 → 再派 → 反复到零模糊 → 写入 wiki/经验/
- 定义：知识复利引擎的传感器——子 Agent 用积木体系主动回忆知识，发现被压缩掉的推导。不是文件修复工具。
- 🛑门禁：G66（跳过循环）、G67（谄媚崩塌）、G68（粉红大象）、G69（自检）、G70（旧路径阻断）
- 例子：主 Agent 写积木 → delegate_task → 子 Agent 回 ❌ 5 个模糊点 → 主 Agent 只修这 5 个 → 再派 → ✅ 零模糊 → 写入 wiki/经验/
- 规则：每轮只修标注的模糊点（不越界）；子 Agent 必须带证据（光说 PASS = FAIL）；≤5 轮，超了标记 dirty 通知晴；零模糊后才写入 wiki/经验/
- 步骤：步骤段直接包含完整 dispatch prompt（见 A.1），主 Agent 加载 skill 后复制到 delegate_task goal 即可派遣子 Agent，不依赖外部文件
- 配置：轮次上限 5；单个模糊点预期修复行数 ≤3；子 Agent 必须输出 ✅/❌ + 模糊点数量

### A.6 异常场景与降级策略

**G68 无 Git 环境。** `git diff --stat` 不可用时，降级为对比 write_file 的 content 行数或 edit 的 oldText/newText 行数差。精度下降但不阻断流程——粉红大象检测从精确 diff 退化为行数估算。

**G67 delegate 消息识别失败。** Hermes 的 delegate_task 返回格式变化时，正则匹配 `'delegate' in role` 或 `'TASK' in content[:200]` 提供保守识别。误报代价低于漏报——非 delegate 消息被误判为 delegate 只是多跑一次匹配，漏报则整个谄媚检测失效。

**Hook 自身故障。** hook_state.py 的轮询文件锁在 Windows 上替代 fcntl——100ms 超时后重试。hook_state.json 读写异常时所有 Hook 降级为放行（不挡正常流程），异常记录到日志。G69 自检是最后一层兜底——Hook 坏了不是静默失效，on_done 时注入警告。

**子 Agent 报告 0 个模糊点但无 ✅ 标记。** G67 检查 has_explicit_marker 而非 fuzziness_count。0 个模糊点 + 无 ✅ = G67 触发重派。显式标记是格式要求——"0 个模糊点但忘了写 ✅"和"真的零模糊"在格式上不可区分，子 Agent 必须学习输出格式。

**主 Agent 连续触发 G68（每次修复都超 3 倍行数）。** 三次以上连续 over_edit_flagged → G69 在 on_done 时检查 edit_scope 的连续标记 → 注入警告："主 Agent 反复过度修复，可能存在粉红大象效应——'修改'这个词在概率空间中被强化。建议降低修复权限或切换为纯报告模式。"

### A.7 与 cognition_loop 的集成点

对抗性验证循环当前独立于 cognition_loop 运行——是手动版的 on-done 验证。未来集成点：
