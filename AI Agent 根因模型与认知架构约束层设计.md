## 一、定义：LLM 的本质

**LLM 是 next-token 概率机，不是任务执行器。** 
每次推理都是从上下文窗口从零开始的概率采样——给它一段文字，它算下一个词最可能是什么。不是"理解"了这段文字，是算概率。每次推理独立：上一轮说过的话、做过的事，下一轮不自动记得。"规则"在它那里不存在——它看到的全是数字，"绝对不能"和"可能可以"只是不同的数字排列，没有轻重之分。

**三个直接后果：无状态、无记忆、token 无强制力。**
它不知道自己在做任务的第几步——你告诉它"现在是第三步"，它读了；你没告诉它，它不知道。上一轮犯的错，下一轮不会自动学到，除非你把错误写进上下文塞给它。"禁止"两个字和"主营业务收入"四个字在它内部是同一类东西——都是向量。这就是提示词注入的根因——它分不出哪句话是系统指令、哪句话是用户数据。这三条不是 Agent 不认真，是架构级的事实。后文列出的所有 Agent 失败模式——走捷径、跳门禁、忘了规则、同意错误建议——全是从这三条长出来的。

**更根本的：LLM 的概率驱动下，最省力的路径恰好概率最高。Agent 不是在偷懒——
是走捷径就是它最自然的选择。根因在架构层，解法也必须到架构层。** 
两条路——剥夺选择权、裁剪输入。后果不可逆的操作，不让 LLM 参与决策：代码在操作前直接拦住，备份没做就拒绝写文件，借贷不平就拒绝提交。后果可修正的操作，让 LLM 参与但控制它看到什么：不是把全部知识扔给它，是每次只给当前这一步需要的积木，按熟练度调整注入量。两条路的共同点——错误的选项从来就没进入 LLM 的概率空间。一条在采样前截断，一条在采样前裁剪。

**两层各管一层人脑机制。** 
LLM 模拟突触——分布式、模糊、擅长模式识别（"这条发票像不像进项"）。积木模拟长期记忆——离散、精确、可审计（"暂估冲回必须在新增进项之前"）。单靠 LLM，规则被洗进分布式参数，改不了、审不了、复用了也不知道。单靠积木，格式稍微变化就不认识，因为积木不会泛化。两层各管一层、互相补位——和人脑一样。




## 二、例子：10 种 Agent 失败模式

LLM 底层本质的 10 种投影——走最短路径 → 事后自知 → 指令边界外溢（该做的不做 → 做的时候不知道不对 → 不该做的做了），每种对应一个具体的工程机制：

**Satisficing（走最短路径）**：概率最高的输出路径恰好是"直接动手"而非"先跑检查"。因为训练数据中行动指令的概率高于声明性前置检查。解法：Hook 在工具调用前强制备份——Hook 是运行时的确定性拦截，不走 LLM 的采样本性。

**事后自知（Temporal Inconsistency）**：同一个 Agent，动手时把内容精简到不完整，事后审视时承认"太精简了，看不懂"。不是嘴硬——动手瞬间和事后审视是两次独立推理，LLM 没有状态记忆，两个采样时刻互相看不见。动手时的概率偏向"简洁一致"，审视时的概率偏向"信息完整"。它事后真知道自己刚才错了，但动手的当下停不住。解法：写文件前系统做结构密度检查——行数和关键结构项数低于基准则拒绝写入，不让第一次采样的结果落地。

**指令边界外溢（Scope Creep）**：LLM 的概率空间天然倾向于"一致性"而非"精确性"——训练数据中全文润色的样本远多于逐行精确修改的样本。你让它改第三段，它可能顺手把第四段也压缩了一行。不是故意越权——在它看来，"把文章整体调整一致"比"只改指定区域、其他一字不动"更自然。解法：Hook 在写文件前对比改动前后的 diff——超出指定范围的变更直接拒绝执行。代码的精确边界补上了 token 层缺失的边界意识。

**二选一思维惯性**：自回归模型逐 token 选最优，训练数据中"选一个方案"比"两个都要"常见得多。解法：上下文组装环节同时注入主动检查（checklist 逐项验证 + git revert 事后检测）和被动检查（正则扫描积木引用 + 实体链接匹配概念名）、置信度（历史验证成功率）和质疑度（异常信号加权分）——两两配对，每条路径独立验证，不留给 LLM 选一个的可能。

**注意力衰减（Lost in the Middle）**：RoPE 位置编码产生 U 型注意力偏见——上下文开头和末尾的 token 权重远高于中间。这是架构级缺陷，不是 Agent 不认真。解法：每次 LLM 调用前，Hook 将系统规则（我是谁→边界在哪→什么能做→管哪些→当前状态）注入到上下文最开头——利用 Transformer 对起始位置有最高注意力权重这一特性。

**token 无强制力**：注意力机制的输入全是无差别的向量。"禁止"和"主营业务收入"在模型内部完全平等。提示词注入就是这个漏洞的极端——模型区分不了系统指令和用户数据。解法：Hook 在代码层用正则匹配文件路径——命中知识库路径则检查操作者是否已获得人工确认，未确认则直接拒绝执行。代码的 if/else 接管了 token 的 "可能/也可能不"。

**粉红大象效应（禁止反而激活）**：负面约束在 token 层面被反复提及，反而强化了被禁行为在概率空间中的权重。神经元层面的激活扩散和概念语义的交叉引用共同导致"越不让想大象，脑子里全是大象"。解法：cognition_loop 的四种能量模式——思维（新任务：全邻域+置信度）、引导（做错/阻断：2-3备选+系统推荐）、肌肉记忆（连续正确≥3次：只推最优）、学习（模拟推演：系统提供地形）——做错时自动从肌肉记忆降级到引导模式，只呈现推荐路径和有限备选，被禁止的操作不在可选列表中。粉红大象的解法不是"不说大象"——是在标红错误路径的同时打绿正确路径，让 Agent 的注意力在绿灯上。

**谄媚崩塌（Sycophantic Collapse）**：训练数据中"同意用户"的序列概率远高于"质疑用户"。LLM 天然倾向于确认而非挑战。解法：操作者/审核程序/探针程序的纠正触发四级连锁：被纠正的工具标记为"待验证"（不再直接信任） → 被它引用的知识条目逐条复查语义是否和代码一致 → 同知识域内的其他相关工具也批量标记待验证 → 该场景下的信任递增步长永久降低。人工或程序的独立信号打破了 LLM 默认同意用户的惯性。

**CoT 损害指令遵循**：推理链越长，初始约束衰减越严重。每步推理都在概率空间中走远——走远了就忘了起点是什么。解法：在 Agent 开始规划之前，系统已将当前步骤需要的全部知识条目注入上下文——不等它遇到困惑再检索，拿到任务时约束已经在眼前。分解粒度锚点通过知识库全文索引检索已有的知识条目标题作为参照，让模型的分解粒度对齐到已有知识的粒度——以标题为锚，不过度拆分。

**Stroop 效应（冲突信号崩溃）**：当互斥指令在上下文中并存时，LLM 无法弃一选一——和人类 Stroop 测试表现一致。解法：每步执行时只拿到当前这一步需要的全部信息（7 种知识类型同时展平），不被告知下一步做什么。所有限制条件（必须过的检查→不能破的规则→上次的踩坑案例）固定排在这步的具体操作说明之前——LLM 先看到边界再看到路径。

## 三、规则：根因 → 硬约束/精密层

### 两种模式

10 种现象归因为两个模式，两者是同一操作的两个角度——不让 LLM 看到它不该选的选项：

**模式 A：剥夺选择权。** 不可逆操作不走 LLM 的 token 采样——分两个拦截点。操作前：工具调用前系统用代码判断——备份未做则拒绝、借贷不平则拒绝、未获确认则拒绝。落地前：LLM 已生成输出但尚未写入——结构密度检查拦截事后自知式的过度精简，diff 对比拦截指令边界外溢式的越权修改。Agent 没有参与权——在它看到工具输出或写入结果之前，代码已经决定了是否放行。

**模式 B：裁剪输入。** 可逆操作走 LLM 但限制概率空间——每次 LLM 思考前系统自动注入当前步骤所需的全部知识条目（7 种 role：编排→定义→规则→例子→步骤→配置→映射），按熟练度控制注入量（新任务=全量知识+置信度标注、做错过=2-3 备选+系统推荐、连续正确=只给最优路径）。

两者的统一逻辑：错误的选项从来就没进入 LLM 的概率空间。A 在采样后截断——代码在操作前或落地前拦住；B 在采样前裁剪——系统不放全量知识，只放当前步骤所需的条目。

### 100% 硬约束层 与 精密注意力层

LLM 没有"必须"和"建议"的区分。整个认知架构的分界线在于：凡是操作后果不可逆的走代码层，凡是操作后果可修正的走注意力层。

**硬约束层（100%）**：备份、门禁、Hook 拦截——对正确性要求绝对。G48 未执行不能写文件，G52 借贷不平衡不能提交，G54 晴未确认不能改知识库。Agent 被排除在决策外——没商量。

**精密注意力层**：上下文组装阶段注入当前步骤所需知识条目，并按熟练度控制注入量——对正确性不要求绝对。Agent 少看一条规则不会立即出错，下次操作完成后的自动检查环节纠正回来。精度通过认知循环迭代收敛：每次执行训练下一次注入的精度——影响范围检查处理直接邻居 → 定时任务全量约束交叉验证（积木的声明 vs 代码的实际调用，分钟级覆盖全局） → 写回可信度评分/节点状态。当前受限于模型能力，但机制本身没有理论上限，认知循环的收敛是经验收敛（靠迭代试错逼近），不是数学收敛（靠梯度保证），愿景是趋近99%。

### 两层必须独立

Cognitive Decay (2026) 的核心警告——同一衰减链上的层不能互相修复。硬约束层只消费 工具调用前代码拦截的判定，精密层只消费 系统从知识库提取并组装到上下文的知识条目。跨层交叉验证只在定时任务做全量数据比对时执行——根因层标记 broken 但原因层没有对应 error 记录的是防护缺口，原因层有高频同类型 error 但根因层没标记 stale 的是行为模式已固化但系统未识别。

### Agent + 积木的认知架构

Agent + 积木的认知架构是对人脑两种记忆系统的工程模拟——神经网络模拟突触的分布式模糊模式识别，积木模拟长期记忆的离散精确规则存储。二者各管一层：LLM 处理模式识别（"这条发票像不像进项"），积木存储精确规则（"暂估冲回必须在新增进项之前"）。单靠神经网络，规则被洗进分布式参数无法审计、无法修正、无法复用。单靠积木，没有泛化能力的系统处理不了格式变化但业务逻辑不变的情况。

### 🛑 距理论的距离（gate）

**软硬约束不同步**：AGENTS.md 7 条铁则只有 3 条有 工具调用前代码拦截对应。⛔1(G48)、⛔2(G54)、⛔7(根目录保护) 有 Hook 拦截。⛔3(新发现写经验)、⛔4(不确定说不确定)、⛔5(出方案等确认)、⛔6(maker≠checker) 在 LLM 眼里和普通文本没有区别。缓解方向：⛔3 可在操作完成后的自动检查环节的 完成判定 中增加检测——本轮有知识操作但没有 知识引擎写入 调用则标记警告。

**终端操作无 Hook 防护**：工具拦截当前只覆盖写文件和改文件操作。终端命令行工具可执行 `rm`、`git push --force`、`mv` 等破坏性命令不被拦截。Phase 8.4 的 git revert 惊醒 Hook 是事后检测——发现错误才降级熟练度等级=guided，不是事前拦截。缓解方向：pre_tool_call Hook 扩展终端命令的路径和命令匹配。

**三层数据流未独立**：错误历史检索（原因层）、约束交叉验证（根因层）、质疑度（Agent 根因层）共享知识库全文索引的查询结果。一个环节的查询误差会滚雪球——错误历史推了什么给 Agent，约束交叉验证就验证什么，质疑度就触发什么。如果全文索引本身有索引偏差，三层同时偏。缓解方向：三层各自独立数据源——原因层用会话事件的 JSON 日志、根因层用拓扑图数据库的边可信度表、Agent 根因层用认知快照中历史熟练度等级的分布。三层在定时任务做全量数据比对时交叉验证。

因果链、工程方法论、Agent 根因是同一问题的三个视角。共享数据源时就区分不了假因果和真因果——三层独立数据源不是性能优化，是让三个视角能互相证伪。

## 四、映射

### AI 训练与积木累积的同构性

LLM 的训练循环和 cognition_loop 的认知循环在算法层面等价——两者都是"从前一步的输入产生输出，用输出与期望的偏差修正下一次的输入"：

LLM 训练：前向传播 → 计算损失 → 反向传播 → 更新网络权重
认知循环：上下文注入 → Agent 执行 → 操作完成后的自动检查环节记录 gap/mode/cause → 写入积木知识条目元数据中的可信度评分/节点状态/经由谁确认的可信度标注 → 下次任务的知识推送策略被更新

两者都从反馈中学习，用下一次的输入控制下一次的行为。区别在于 LLM 改的是网络权重（分布式、模糊、不可审计），积木系统改的是上下文注入策略（离散、精确、可追溯）。二者的模拟粒度不同——神经网络模拟突触，积木模拟长期记忆。不是谁更好，是管不同的认知层。

这个架构不是某一个系统特有的设计。它是 LLM 作为 next-token 概率机这一事实在工程上唯一自洽的回应：凡是 LLM 不该参与的决定，剥夺选择权；凡是 LLM 需要参与但不该自由发挥的，裁剪输入。训练循环和认知循环在算法层面的等价性，不是巧合——是同一个"用偏差修正下一次输入"的模式在两个不同粒度上的实现。
