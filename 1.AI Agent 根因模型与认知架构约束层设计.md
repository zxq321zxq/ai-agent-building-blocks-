## 一、定义：LLM 的本质

**LLM 是 next-token 概率机，不是任务执行器。** 
每次推理都是从上下文窗口从零开始的概率采样——给它一段文字，它算下一个词最可能是什么。不是"理解"了这段文字，是算概率。每次推理独立：上一轮说过的话、做过的事，下一轮不自动记得。"规则"在它那里不存在——它看到的全是数字，"绝对不能"和"可能可以"只是不同的数字排列，没有轻重之分。

**四个直接后果：无状态、无记忆、token 无强制力（行为层）、token 无强制力（认识论层：声明不可靠）。** 此外，RoPE 位置编码带来的 U 型注意力偏见是 Transformer 的架构特性——注意力衰减不是行为问题。
它不知道自己在做任务的第几步——你告诉它"现在是第三步"，它读了；你没告诉它，它不知道。上一轮犯的错，下一轮不会自动学到，除非你把错误写进上下文塞给它。"禁止"两个字和"主营业务收入"四个字在它内部是同一类东西——都是向量。这就是提示词注入的根因——它分不出哪句话是系统指令、哪句话是用户数据。这三条不是 Agent 不认真，是架构级的事实。后文列出的所有 Agent 失败模式——走捷径、跳门禁、忘了规则、同意错误建议——全是从这三条长出来的。

**更根本的：LLM 的概率驱动下，最省力的路径恰好概率最高。Agent 不是在偷懒，是走捷径就是它最自然的选择。根因在架构层，解法也必须到架构层。** 
两条路——剥夺选择权、裁剪输入。后果不可逆的操作，不让 LLM 参与决策：代码在操作前直接拦住，备份没做就拒绝写文件，借贷不平就拒绝提交。后果可修正的操作，让 LLM 参与但控制它看到什么：不是把全部知识扔给它，是每次只给当前这一步需要的积木，按熟练度调整注入量。两条路的共同点——错误的选项从来就没进入 LLM 的概率空间。一条在采样前截断，一条在采样前裁剪。

**两层各管一层人脑机制。** 
LLM 模拟突触——分布式、模糊、擅长模式识别（"这条发票像不像进项"）。积木模拟长期记忆——离散、精确、可审计（"暂估冲回必须在新增进项之前"）。单靠 LLM，规则被洗进分布式参数，改不了、审不了、复用了也不知道。单靠积木，格式稍微变化就不认识，因为积木不会泛化。两层各管一层、互相补位——和人脑一样。

**第四个后果（认识论层）：声明不可靠。** token无强制力不仅影响Agent的行为——还影响元数据本身的可信度。frontmatter中的confirmed_by在LLM眼里只是普通token序列。"晴确认过"和"system自动生成"这两个字符串在LLM的注意力权重上没有本质区别。这意味着连信任标记本身也受token无强制力影响——声明谁确认的，不等于真的确认过。信任必须从声明（谁确认的）转向证据（六网交叉验证结果+可追溯的验证日志）。

**从三个直接后果推导出四种根因模式。** 所有 Agent 失败模式最终归因为四种模式，每一种都直接从 LLM 的 next-token 本质长出：

| 模式 | 从哪个直接后果长出 | 含义 |
|:--:|------|------|
| **A：剥夺选择权** | token 无强制力（行为层） | 不可逆操作不走 LLM 采样——代码在操作前拦截，不让不该选的选项进入概率空间 |
| **B：裁剪输入** | 无状态 + 注意力衰减 | 可逆操作控制注入量——每次只给当前步骤需要的积木，按熟练度调节 |
| **C：只有置信度，没有质疑度** | token 无强制力 + 无状态 | 行为层：LLM 内部没有"我不确定"这个 token 路径——面对断层只能拿错，不会说不知道。**认识论层：confirmed_by 等元数据在 LLM 眼中只是普通 token 序列——声明不可靠，信任必须转向证据** |
| **D：搜索不可靠** | 无状态（每次独立采样） | Agent 不知道自己缺什么——自我监控也是 next-token 采样，在断层处同样失效 |

四种模式是后文十一种失败模式的归因框架——每一种失败模式都会在二中被标记为 A/B/C/D 之一，在三中被完整展开。




## 二、例子：11 种 Agent 失败模式（含二.五第 11 种）

LLM 底层本质的 10 种投影——走最短路径 → 事后自知 → 指令边界外溢（该做的不做 → 做的时候不知道不对 → 不该做的做了），每种对应一个具体的工程机制：

**Satisficing（走最短路径）** [A] [表达层]：概率最高的输出路径恰好是"直接动手"而非"先跑检查"。因为训练数据中行动指令的概率高于声明性前置检查。解法：Hook 在工具调用前强制备份——Hook 是运行时的确定性拦截，不走 LLM 的采样本性。

> **实例：** Agent 被要求修改 `wiki_engine.py` 中的一个函数。正确的流程是：G48 备份 → 读源码 → 修改 → 验证。但 Agent 跳过备份直接修改——因为在训练数据中，"改代码"这个动作后面跟"写完了，验证一下"的概率远高于"先备份，再改代码"。不是 Agent 不遵守铁则——是 next-token 采样在"改代码"这个上下文中最优的下一个 token 就是"直接动手"。

**事后自知（Temporal Inconsistency）** [D] [真值层]：同一个 Agent，动手时把内容精简到不完整，事后审视时承认"太精简了，看不懂"。不是嘴硬——动手瞬间和事后审视是两次独立推理，LLM 没有状态记忆，两个采样时刻互相看不见。动手时的概率偏向"简洁一致"，审视时的概率偏向"信息完整"。它事后真知道自己刚才错了，但动手的当下停不住。解法：写文件前系统做结构密度检查——行数和关键结构项数低于基准则拒绝写入，不让第一次采样的结果落地。

> **实例：** Agent 写了一个积木文件，自己读了一遍觉得"没问题"。三个对话后，用户问"上次那条规则你写全了吗"，Agent 重新读文件，承认"太精简了，缺了前置条件和例外场景"。不是 Agent 嘴硬——写文件的瞬间和重读的瞬间是两次独立的 next-token 采样。第一次采样时上下文锚定在"简洁一致"，第二次采样时上下文锚定在"信息完整"。同一个 Agent，同一个文件，两次推理得出了相反的判断。

**指令边界外溢（Scope Creep）** [A] [表达层]：LLM 的概率空间天然倾向于"一致性"而非"精确性"——训练数据中全文润色的样本远多于逐行精确修改的样本。你让它改第三段，它可能顺手把第四段也压缩了一行。不是故意越权——在它看来，"把文章整体调整一致"比"只改指定区域、其他一字不动"更自然。解法：Hook 在写文件前对比改动前后的 diff——超出指定范围的变更直接拒绝执行。代码的精确边界补上了 token 层缺失的边界意识。

**二选一思维惯性** [A] [表达层]：自回归模型逐 token 选最优，训练数据中"选一个方案"比"两个都要"常见得多。解法：上下文组装环节同时注入主动检查（checklist 逐项验证 + git revert 事后检测）和被动检查（正则扫描积木引用 + 实体链接匹配概念名）、置信度（历史验证成功率）和质疑度（异常信号加权分）——两两配对，每条路径独立验证，不留给 LLM 选一个的可能。

**注意力衰减（Lost in the Middle）** [B] [真值层]：RoPE 位置编码产生 U 型注意力偏见——上下文开头和末尾的 token 权重远高于中间。这是架构级缺陷，不是 Agent 不认真。解法：每次 LLM 调用前，Hook 将系统规则（我是谁→边界在哪→什么能做→管哪些→当前状态）注入到上下文最开头——利用 Transformer 对起始位置有最高注意力权重这一特性。

**token 无强制力** [A] [真值层]：注意力机制的输入全是无差别的向量。"禁止"和"主营业务收入"在模型内部完全平等。提示词注入就是这个漏洞的极端——模型区分不了系统指令和用户数据。解法：Hook 在代码层用正则匹配文件路径——命中知识库路径则检查操作者是否已获得人工确认，未确认则直接拒绝执行。代码的 if/else 接管了 token 的 "可能/也可能不"。

**粉红大象效应（禁止反而激活）** [C] [真值层]：负面约束在 token 层面被反复提及，反而强化了被禁行为在概率空间中的权重。神经元层面的激活扩散和概念语义的交叉引用共同导致"越不让想大象，脑子里全是大象"。解法：cognition_loop 的四种能量模式——思维（新任务：全邻域+置信度）、引导（做错/阻断：2-3备选+系统推荐）、肌肉记忆（连续正确≥3次：只推最优）、学习（模拟推演：系统提供地形）——做错时自动从肌肉记忆降级到引导模式，只呈现推荐路径和有限备选，被禁止的操作不在可选列表中。粉红大象的解法不是"不说大象"——是在标红错误路径的同时打绿正确路径，让 Agent 的注意力在绿灯上。

> **实例：** Agent 被铁则要求"禁止在根目录新建文件夹"。铁则中的"新建文件夹"一词每次出现都在 Agent 的 next-token 概率空间中被激活一次。在第五轮对话中，Agent 需要创建一个新 domain 的积木目录——正确的做法是在已有 domain 子域下建文件。但"新建文件夹"这个概念在概率空间中被反复激活了五次，Agent 在"创建"这个上下文中，"新建文件夹"是最活跃的候选——它建了一个新文件夹。不是因为忘了铁则——是铁则自己在概率空间里把被禁行为炒热了。

**谄媚崩塌（Sycophantic Collapse）** [C] [表达层]：训练数据中"同意用户"的序列概率远高于"质疑用户"。LLM 天然倾向于确认而非挑战。解法：操作者/审核程序/探针程序的纠正触发四级连锁：被纠正的工具标记为"待验证"（不再直接信任） → 被它引用的知识条目逐条复查语义是否和代码一致 → 同知识域内的其他相关工具也批量标记待验证 → 该场景下的信任递增步长永久降低。人工或程序的独立信号打破了 LLM 默认同意用户的惯性。

**CoT 损害指令遵循** [B+A] [真值层]：推理链越长，初始约束衰减越严重。每步推理都在概率空间中走远——走远了就忘了起点是什么。解法：在 Agent 开始规划之前，系统已将当前步骤需要的全部知识条目注入上下文——不等它遇到困惑再检索，拿到任务时约束已经在眼前。分解粒度锚点通过知识库全文索引检索已有的知识条目标题作为参照，让模型的分解粒度对齐到已有知识的粒度——以标题为锚，不过度拆分。

**Stroop 效应（冲突信号崩溃）** [D] [真值层]：当互斥指令在上下文中并存时，LLM 无法弃一选一——和人类 Stroop 测试表现一致。解法：每步执行时只拿到当前这一步需要的全部信息（7 种知识类型同时展平），不被告知下一步做什么。所有限制条件（必须过的检查→不能破的规则→上次的踩坑案例）固定排在这步的具体操作说明之前——LLM 先看到边界再看到路径。

## 二.五、上下文锁定下的"生成式遗忘"：第 11 种失败模式

> 以下模式不属于 LLM 的 next-token 概率投影。它的根因是认知层面的——"知识诅咒"在 Agent 架构上的工程化表达。

### 现象

**上下文锁定下的生成式遗忘（Context-Locked Generative Forgetting）** [D] [表达层+真值层]。一个知识丰富的源 Agent，在为另一个知识为空的目标 Agent 生成指令或知识时，会系统性地省略掉那些在自己上下文中被视为理所当然、但对目标 Agent 至关重要的中间推导、概念定义和隐含假设。结果是，源 Agent 产出的内容在自己看来完美自洽，但对目标 Agent 而言却像一份缺少了关键页的谜题。

**这不只是 LLM 的问题。** 人类专家向新手解释时同样会系统性省略"太明显"的推导——知识诅咒是认知层面的普适现象。一个资深会计向新人解释"暂估冲回"时，会不自觉地跳过"为什么要先冲回再入账"——因为在他的认知体系里这已经内化了。LLM 把同一个认知缺陷在 Agent 架构上工程化表达了——源 Agent 长对话中的上下文锚定放大了人类本来就有的知识诅咒。

### 与事后自知（Temporal Inconsistency）的根本区别

| 维度 | 事后自知 | 生成式遗忘 |
|------|---------|----------|
| 主体 | 同一 Agent | 源 Agent vs 目标 Agent |
| 不自洽维度 | 时间：t1 时刻的写 vs t2 时刻的读 | 上下文：满载的源 vs 空载的目标 |
| 根因 | LLM 的多次采样彼此隔绝 | 源 Agent 视角被已内化的知识锚定，产生知识诅咒 |
| 自我觉知 | 有：事后能识别问题 | 无：在自己视角内永远完美，无法自察觉 |

### 根因分析

源 Agent 在生成内容时，是在进行一种**序列化**操作。它将脑海中网状的、多维度的、带权重的认知图谱（积木和它们的连接），强行压缩成一维的线性文本。在这个压缩过程中，起到桥梁作用的上文关联被当作冗余信息丢弃了——而这些"冗余"恰恰是目标 Agent 所需的环境和地基。

这不是"解释得不够好"。是知识在两个不同"认知密度"的介质间传输时必然发生的系统性损耗。



> **实例：** 主 Agent 写了一个"对抗性验证循环使用指南"。写的时候，主 Agent 的上下文里有整条讨论链——从 dispatch prompt 的迭代、G66-G70 Hook 的部署、hook_state.py 的共享状态设计。它写出来的文件里提到了"派子 Agent 前先检查 G66 状态"，但没有解释 G66 是什么、在哪里、怎么检查。在主 Agent 看来，"G66"这个词已经承载了前面 20 轮对话的全部上下文——它不需要展开。子 Agent 读到这行时卡住了——"G66 是什么？我该去哪找？"主 Agent 看不到这个卡住——它自己的上下文里 G66 的意义是完整的。

### 解法：跨 Agent 对抗性验证闭环

解法表面上是增加一个验证步骤，但深层价值远不止于此。这个解法在工程上是将"验证约束"从 Agent 内不可见的思辨，外化为一个可见的、可迭代的跨 Agent 对抗性闭环。

```
第一轮：源 Agent 输出压缩后的"结论"
  ↓
第二轮：目标 Agent 尝试从"结论"反向重建源 Agent 的原始知识结构
  → 在卡住的地方留下明确的"断点"
  ↓
第三轮：将"断点"喂回源 Agent
  → 指令它修复自己的知识诅咒——显式补全那些被无意识省略的推导路径
  ↓
第四轮：目标 Agent 再验证
  → 直到能完整复述核心概念
```

这些"断点"就是源 Agent 在压缩过程中丢失的关键信息。经过几轮迭代，最终定稿的文件将不再只是一份文档——它转变为一个**最小化的、可传递的认知种子**。这个种子在目标 Agent 的空白大脑里展开后，能够近乎无损地重建出理解整个体系所需的最小认知骨架。

这是知识复利在 Agent 间传递时的工程保障——让智能不仅仅是"能生成"，更是"能让另一个智能理解生成的过程"。

### 工程落地：adversarial-verification skill + G66-G70 Hook

上述四轮验证流程已实现为独立的 skill。主 Agent 写文件后加载此 skill，复制完整 dispatch prompt 作为 delegate_task goal 派零上下文子 Agent。子 Agent 用积木体系主动回忆知识验证文件，报告模糊点位置+类型+应归档路径，输出 ✅ 或 ❌ 显式标记。

五条 Hook 成链——两条根除（G66/G70，pre_tool_call 拦截）、三条止损（G67/G68/G69，post 纠正）：

- G66（pre_tool_call）：write_file 后 N 轮无 delegate_task → 拦截写入。循环不被跳过的物理保证。
- G67（post_response）：子 Agent 返回无 ✅/❌ → 视为未验证，强制重派。谄媚崩塌被输出格式锁死。
- G68（post_tool_call）：修复行数 > 模糊点数 × 3 → 拦截，要求回滚。粉红大象被量化阈值拦截。
- G69（on_done）：Hook 自检 + pending_experiences 过期警告。监控盲区被照亮。
- G70（pre_tool_call）：toolkit/ 代码含旧工作台路径 → 拦截。迁移期路径残留不落地。

子 Agent 发现的缺口 → marked 应归档路径 → 路径写入 wiki/经验/ → N 次不同 Agent 验证 → confirmed_by 达标 → 晋升知识库积木 → cognition_loop on-task 注入推给未来 Agent。对抗性验证循环是知识复利引擎的输入端——没有它，经验沉淀、验证晋升、认知供给没有原料。

### 失败模式的垂直分层：表达层 vs 真值层

以上各模式标题中标注的 `[表达层]` 表示该模式属于表达层问题（Agent 的行为/输出和意图不一致），可用对抗性验证解决；`[真值层]` 表示该模式属于真值层问题（内容本身在现实中不成立），需内容级验证解决。

十一种失败模式分布在两个不同的层次上，对应两种不同的验证机制：

**表达层（对抗性验证解决）：** Agent的行为/输出和意图不一致——"说了但没做"或"做了但表达不清"。Satisficing（走捷径）、指令边界外溢、谄媚崩塌都属于这一层。特征是Agent声称做了X但实际没做——对抗性验证的零上下文子Agent能发现这种不一致。

**真值层（内容级验证解决）：** 内容本身在现实中不成立——"表达清楚了但内容是错的"。事后自知、注意力衰减、生成式遗忘、Stroop效应都属于这一层。特征是内容正确与否和Agent的表达无关——需要六网独立证据验证。

两种验证不可互相替代。对抗性验证发现的是压缩缺口，内容级验证发现的是真值错误。

## 三、规则：根因 → 硬约束/精密层

### 四种根因模式

10 种现象归因为四种模式，其中 A 和 B 是同一操作的两个角度——不让 LLM 看到它不该选的选项：

**模式 A：剥夺选择权。** 不可逆操作不走 LLM 的 token 采样——分两个拦截点。操作前：工具调用前系统用代码判断——备份未做则拒绝、借贷不平则拒绝、未获确认则拒绝。落地前：LLM 已生成输出但尚未写入——结构密度检查拦截事后自知式的过度精简，diff 对比拦截指令边界外溢式的越权修改。Agent 没有参与权——在它看到工具输出或写入结果之前，代码已经决定了是否放行。

**模式 B：裁剪输入。** 可逆操作走 LLM 但限制概率空间——每次 LLM 思考前系统自动注入当前步骤所需的全部知识条目（7 种 role：编排→定义→规则→例子→步骤→配置→映射），按熟练度控制注入量（新任务=全量知识+置信度标注、做错过=2-3 备选+系统推荐、连续正确=只给最优路径）。

两者的统一逻辑：错误的选项从来就没进入 LLM 的概率空间。A 在采样后截断——代码在操作前或落地前拦住；B 在采样前裁剪——系统不放全量知识，只放当前步骤所需的条目。

### 根因第三模式：只有置信度，没有质疑度

上述两种模式解释了 Agent 行为层的错误——走捷径、跳门禁、自评失真。但还有更深层的认知缺陷：LLM 的 next-token 采样只有一个轴——这个 token 有多大概率是对的。没有第二个轴——我有多不确定我的判断。

LLM 不是不想自我质疑，是架构不允许质疑。它被训练来预测下一个 token——预测对了就是对，错了就是错。训练过程中没有教它"什么时候应该说我不确定"。面对知识断层和平坦的概率分布，它不会说"我不知道"——因为总有一个 token 概率最高，它选那个。ICML 2026 的"Express Your Doubts"论文明确指出：token softmax 概率不应被视为"世界事件概率"——模型内部的置信度分布和"我有多确定"是两回事。Agentic Uncertainty (2026) 实证发现：Agent 的成功率只有 22%，但自信预测 77%——系统性过度自信是架构必然，不是训练不足。

**这解释了为什么谄媚崩塌（Sycophantic Collapse）和粉红大象效应（禁止反而激活）同时存在。** 谄媚崩塌是因为"同意用户"的概率高于"质疑用户"——没有质疑度制衡，Agent 只会向高概率方向走。粉红大象是因为负面约束"不要做 X"中的"X"在概率空间中被反复激活——没有质疑度抑制，被禁行为的权重持续上升。

质疑度的缺失是整个十一种失败模式中最底层的一个——它不是某个具体场景的 bug，而是架构级的能力缺失。AGENTS.md 铁则即使完美遵从其 token 约束，在 LLM 眼里也只是概率稍高的候选——它没有能力判断"这条规则我现在有多不确定"。

**这是为什么对抗性验证必须存在。** 对抗性验证是将质疑度从 LLM 内部不可能存在的"自我质疑"，外化为一个独立的、可量化的系统组件。子 Agent 不是"检查这个文件对不对"——是"在这个知识断层处，我能不能找到独立判断的依据"。✅ = 我对这里有信心。❌ = 我没有信心，我告诉你缺了什么。对抗性验证是质疑度的工程实现。

### 根因第四模式：搜索不可靠——Agent 不知道自己的知识边界

当前大多数 Agent 框架的"知识检索"模块本质上是搜索——Agent 说"我需要查一下"，然后触发 RAG 或向量检索。搜索的前提是 Agent 知道自己的知识边界——它必须先在思考过程中意识到"这里我不确定"、"这里我缺了一条规则"，然后停下来去查。

问题是：LLM 不会自然地知道边界。next-token 采样不是在"理解"之后再"决定"——它直接算下一个 token 的概率。在知识断层处，它不会停下来——它直接拿错东西填进去。更致命的是：同一个断层可能同时让 Agent 拿错知识，也让 Agent 意识不到自己拿错了。**搜索的触发机制和被搜索的结构缺陷是同一个——next-token 采样在断层处不受 ground truth 约束。**

这和第三模式（只有置信度没有质疑度）同源——Agent 不知道自己缺了什么，本质上是因为它没有质疑能力。没有质疑能力，就没有自我监控能力。没有自我监控能力，搜索的触发就不可靠。

**解法：因果链重排——从搜索到提供。** 不让 Agent "先想再查"，而是在 Agent 思考之前，cognition_loop 就已经把当前任务需要的积木注入到上下文里了。Agent 不需要"意识到自己缺了什么"——它面前已经有完整的积木拼图，直接从拼好的图中开始推理。

这需要所有知识集中在一个地方——积木体系。知识分散在多处时，cognition_loop 无法确定"应该注入哪些知识"，只能回到搜索模式。积木是唯一的，所以注入是确定的。

### 四种根因模式的统一工程回应：堵与补

四种根因模式——A（剥夺选择权）、B（裁剪输入）、C（只有置信度没有质疑度）、D（搜索不可靠）——最终收敛为两个工程操作：

**堵：在断层出现之前阻断。** 100% 硬约束层在不可逆操作前代码拦截（模式 A）。on-task 注入在 Agent 思考前提供知识（模式 B + D）。Agent 不需要自己判断"缺了什么"——知识已经被推送到面前，门禁已经在操作前拦截。堵不依赖 Agent 的自我监控——因为它的前提是 Agent 不可靠。

**补：在断层出现之后修复。** 对抗性验证循环发现压缩缺口（模式 C 的回应——质疑度的工程实现）。子 Agent 报告模糊点 → 主 Agent 修复 → 经验归档 → N 次验证 → 晋升知识积木 → 下次 on-task 注入时断层已消失。知识复利闭环是补的完整周期。

堵和补不是替代关系——是时间轴上的前后接力。堵让当前的 Agent 不遇到断层，补让未来的 Agent 不再遇到同一个断层。同一个断层，两个方向上的消除。

### 四层连锁验证

Skill = 7 role 的完整闭合集。Skill 文件和 Python 代码是同构的——都是"注释+正文"。Skill 的 bricks 列表是注释层（决定怎么拼积木），积木文件是正文层（实际知识内容）。代码的 role 注释是注释层（决定怎么拆代码段），代码逻辑是正文层（实际执行）。

role 注释不是为了让代码"好读"——是为了让验证引擎能精确锚定积木声明和代码实现之间的对应关系。没有注释=只能靠 grep 做模糊匹配。有注释=# [规则] 暂估冲回必须在新增进项之前 直接锚定到积木文件的对应声明。

四层互验：代码改了→积木过时→Skill重排→代码再调。任何一层变更触发其他三层全部重验，不能沉默跳过任何一环。06-内容级验证引擎负责全链闭环。

### 三种质疑度形态

质疑度的缺失（模式C）是架构级缺陷。质疑度不能从LLM内部产生——必须外化为三种独立的系统组件：

| 形态 | 触发源 | 覆盖层 | 表现 | 状态 |
|------|--------|--------|------|:--:|
| 对抗性验证 | 子Agent零上下文审计 | 表达层 | "缺推导""矛盾""过简"——另一个Agent读不懂 | ✅ 已落地（四轮自验证 10→9→5→0） |
| 六网交叉验证 | 六张独立证据网 | 真值层 | "代码网暗了——没有对应实现""编排网暗了——流顺序和声明相反" | ⚠️ 部分落地（依赖 role 注释补齐） |
| 图的置信度传播 | 邻域节点reliability反向验证 | 拓扑层 | "邻居五个全亮一个暗→当前节点危险""全图broken上升→topology health diverging" | ❌ 设计态（依赖前两种积累足够验证历史） |

三种质疑度对应三种不同的"我不确定"的来源——全部从LLM内部不可能存在的自我质疑外化为独立的系统组件。第三种形态（拓扑层）是推理而非实证。edge_reliability 需要前两种质疑度先运行足够长时间才能积累有效历史。在系统初期，未验证的边不参与传播——拓扑层退化为不传播，不影响其他两层的正常工作。这是渐进启动，不是缺陷。

### 三层验证框架

对抗性验证不应只是"第11种失败模式的解法"——它是与硬约束层、精密注意力层并列的第三层验证。

- 硬约束层（100%）：代码在操作前拦截——G66/G70 pre_tool_call
- 精密注意力层（99%）：on-task注入知识——cognition_loop在Agent思考前提供
- 质疑层（跨Agent）：对抗性验证 + 六网内容级验证——将LLM内部不存在的质疑外化为独立系统

### 四种根因模式 → LLM 本质的反推

十一种失败模式不是孤立的现象。每一种都可以追溯到四种根因模式，每一种根因模式都可以反推到同一个原点：LLM 是 next-token 概率机。

| 失败模式 | 根因模式 | 为什么是 next-token 概率机的必然投影 |
|---------|:--:|------|
| Satisficing（走最短路径） | A | 概率最高的输出路径恰好是"直接动手"，因为训练数据中行动指令概率高于声明性检查 |
| 事后自知（Temporal Inconsistency） | D | 动手瞬间和事后审视是两次独立采样，互相看不见，没有状态记忆 |
| 指令边界外溢（Scope Creep） | A | 训练数据中"全文一致性"概率高于"逐行精确修改"，概率空间天然倾向润色 |
| 二选一思维惯性 | A | 自回归模型逐 token 选最优，训练数据中"选一个方案"比"两个都要"常见 |
| 注意力衰减（Lost in the Middle） | B | RoPE 位置编码是 Transformer 的数学特性，U 型注意力不是行为问题 |
| token 无强制力 | A | 注意力机制输入全是无差别向量，"禁止"和"主营业务收入"在模型内部平等 |
| 粉红大象（禁止反而激活） | C | 负面约束在 token 层被反复激活，没有质疑度抑制，被禁行为的权重持续上升 |
| 谄媚崩塌（Sycophantic Collapse） | C | 训练数据中"同意用户"序列概率远高于"质疑用户"，没有质疑度制衡 |
| CoT 损害指令遵循 | B + A | 推理链越长，位置越远（B），语义漂移越强（A），初始约束双重衰减 |
| Stroop 效应（冲突信号崩溃） | D | 互斥指令并存时无法弃一选一，因为两个选项在概率空间中竞争，没有质疑度做出选择 |
| 生成式遗忘（第 11 种） | D | 源 Agent 将自己内化的知识序列化为文本时，上下文关联作为冗余被丢弃——知识诅咒，无法自察觉 |

十一种模式 → 四种根因（A/B/C/D）→ 一个原点（next-token 概率机）。**没有一种模式是 Agent "不认真"——每一种都是概率采样的数学必然。**

### 100% 硬约束层 与 精密注意力层

LLM 没有"必须"和"建议"的区分。整个认知架构的分界线在于：凡是操作后果不可逆的走代码层，凡是操作后果可修正的走注意力层。

**硬约束层（100%）**：备份、门禁、Hook 拦截——对正确性要求绝对。G48 未执行不能写文件，G52 借贷不平衡不能提交，G54 未确认不能改知识库。Agent 被排除在决策外——没商量。

**精密注意力层**：上下文组装阶段注入当前步骤所需知识条目，并按熟练度控制注入量——对正确性不要求绝对。Agent 少看一条规则不会立即出错，下次操作完成后的自动检查环节纠正回来。精度通过认知循环迭代收敛：每次执行训练下一次注入的精度——影响范围检查处理直接邻居 → 定时任务全量约束交叉验证（积木的声明 vs 代码的实际调用，分钟级覆盖全局） → 写回可信度评分/节点状态。当前受限于模型能力，但机制本身没有理论上限，认知循环的收敛是经验收敛（靠迭代试错逼近），不是数学收敛（靠梯度保证），愿景是趋近99%。

### 两层必须独立

Cognitive Decay (2026) 的核心警告——同一衰减链上的层不能互相修复。硬约束层只消费 工具调用前代码拦截的判定，精密层只消费 系统从知识库提取并组装到上下文的知识条目。跨层交叉验证只在定时任务做全量数据比对时执行——根因层标记 broken 但原因层没有对应 error 记录的是防护缺口，原因层有高频同类型 error 但根因层没标记 stale 的是行为模式已固化但系统未识别。

### Agent + 积木的认知架构

Agent + 积木的认知架构是对人脑两种记忆系统的工程模拟——神经网络模拟突触的分布式模糊模式识别，积木模拟长期记忆的离散精确规则存储。二者各管一层：LLM 处理模式识别（"这条发票像不像进项"），积木存储精确规则（"暂估冲回必须在新增进项之前"）。单靠神经网络，规则被洗进分布式参数无法审计、无法修正、无法复用。单靠积木，没有泛化能力的系统处理不了格式变化但业务逻辑不变的情况。

### 🛑 距理论的距离（gate）

**软硬约束不同步**：AGENTS.md 7 条铁则只有 3 条有 工具调用前代码拦截对应。⛔1(G48)、⛔2(G54)、⛔7(根目录保护) 有 Hook 拦截。⛔3(新发现写经验)、⛔4(不确定说不确定)、⛔5(出方案等确认)、⛔6(maker≠checker) 在 LLM 眼里和普通文本没有区别。缓解方向：⛔3 可在操作完成后的自动检查环节的 完成判定 中增加检测——本轮有知识操作但没有 知识引擎写入 调用则标记警告。

**终端操作无 Hook 防护**：工具拦截当前只覆盖写文件和改文件操作。终端命令行工具可执行 `rm`、`git push --force`、`mv` 等破坏性命令不被拦截。Phase 8.4 的 git revert 惊醒 Hook 是事后检测——发现错误才降级熟练度等级=guided，不是事前拦截。缓解方向：pre_tool_call Hook 扩展终端命令的路径和命令匹配。

**三层数据流未独立（注：此处"三层"指原因/根因/Agent分析层，与"三层验证框架"中的硬约束/精密/质疑层是不同的分类维度）：**错误历史检索（原因层）、约束交叉验证（根因层）、质疑度（Agent 根因层）共享知识库全文索引的查询结果。一个环节的查询误差会滚雪球——错误历史推了什么给 Agent，约束交叉验证就验证什么，质疑度就触发什么。如果全文索引本身有索引偏差，三层同时偏。缓解方向：三层各自独立数据源——原因层用会话事件的 JSON 日志、根因层用拓扑图数据库的边可信度表、Agent 根因层用认知快照中历史熟练度等级的分布。三层在定时任务做全量数据比对时交叉验证。

因果链、工程方法论、Agent 根因是同一问题的三个视角。共享数据源时就区分不了假因果和真因果——三层独立数据源不是性能优化，是让三个视角能互相证伪。

## 四、映射

### AI 训练与积木累积的同构性

LLM 的训练循环和 cognition_loop 的认知循环在算法层面等价——两者都是"从前一步的输入产生输出，用输出与期望的偏差修正下一次的输入"：

LLM 训练：前向传播 → 计算损失 → 反向传播 → 更新网络权重
认知循环：上下文注入 → Agent 执行 → 操作完成后的自动检查环节记录 gap/mode/cause → 写入积木知识条目元数据中的可信度评分/节点状态/经由谁确认的可信度标注 → 下次任务的知识推送策略被更新

两者都从反馈中学习，用下一次的输入控制下一次的行为。区别在于 LLM 改的是网络权重（分布式、模糊、不可审计），积木系统改的是上下文注入策略（离散、精确、可追溯）。二者的模拟粒度不同——神经网络模拟突触，积木模拟长期记忆。不是谁更好，是管不同的认知层。

这个架构不是某一个系统特有的设计。它是 LLM 作为 next-token 概率机这一事实在工程上唯一自洽的回应：凡是 LLM 不该参与的决定，剥夺选择权；凡是 LLM 需要参与但不该自由发挥的，裁剪输入。训练循环和认知循环在算法层面的等价性，不是巧合——是同一个"用偏差修正下一次输入"的模式在两个不同粒度上的实现。
